# 設計の観点

設計（Design Doc 作成、実装、レビュー）に共通する観点集。要件整理・API 設計・モジュール構造・実装レベルの正しさをカバーする。

## 要件・スコープ

- What / Why が明確か
- 要件がテスト可能な形式か（「〜したら〜になる」）
- スコープの境界が明確か（含むもの・含まないもの）
- 要件間に矛盾がないか
- 非機能要件（性能、互換性、セキュリティ）の考慮

## 技術的アプローチ

- 代替案が十分に検討されているか
- 選択の根拠が明確か
- 実現可能性とリスク
- **既存コードベースのパターンと整合しているか**（独自解法を導入するより、既存の factory・helper・ミドルウェア等を再利用できないか）

## API / インターフェース設計（利用者目線）

- インターフェースが直感的か
- 型定義が利用者にとって分かりやすいか（必須 / 任意の区分、命名）
- 命名が一貫しているか（同じパターンに同じ命名規則）
- public / internal の境界が明確か（内部実装が漏れていないか）
- エラーメッセージ・エラーハンドリング方針が利用者にとって有用か
- デフォルト値が妥当か
- カスタマイズ性が過不足ないか
- 公開メソッド・型に doc comment が付与されているか（実装段階）

## モジュール性

- 凝集度: 各モジュールが単一の責務を持つか
- 結合度: 関数 / メソッドが必要最小限のパラメータを受け取るか
- 情報隠蔽: 実装詳細がカプセル化されているか
- 依存方向: 一方向か、循環していないか
- 置換可能性: コンポーネントを差し替えられるか
- 変更伝播: 1 箇所の変更が波及する範囲は妥当か

## パッケージ構成

- ディレクトリが技術的役割（models/、views/、controllers/、machines/ 等）で分かれていないか
- 「同じステートマシン」「同じレイヤー」という理由でグルーピングされていないか
- 各フィーチャーディレクトリが自己完結しているか（他フィーチャーへの直接参照がないか）

**判定方法**: `git diff --name-only` で変更ファイルを取得し、ディレクトリ構造を確認。新規ディレクトリが作られた場合は特に注意。

## ファイル粒度

- 1 クラス = 1 ファイルの過度な分割になっていないか
- 関連するコード（型・イベント・定数・State 等）が 1 ファイルに凝集しているか
- コードの流れを上から読めるか

**判定方法**: 新規作成ファイルの行数と内容を確認。50 行以下のファイルが複数作られている場合は統合を検討。

## リソース管理

- リソース（ファイル、接続、タイマー等）の生成と解放が対になっているか
- dispose / cleanup の全パスで漏れがないか
- 外部から注入されたリソースを勝手に閉じていないか
- キャッシュの上限と eviction が制御されているか

## 正当性

- 外部状態（乱数、時刻等）への意図しない依存がないか
- 数値の境界条件（オーバーフロー、浮動小数点誤差、ゼロ除算）
- 状態遷移の全パスで整合性が保たれているか
- マジックナンバーがないか

## 型安全性

- `as` キャストや `any` で型安全性を迂回していないか
- union 型の消費側で適切に判別されているか
- ランタイムでの引数不足・型不一致の可能性

## 実装と設計の整合性（design.md がある場合）

- design.md の要件が実装されているか
- design.md にない機能が追加されていないか
- インターフェースが design.md の定義と一致しているか
