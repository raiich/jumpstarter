# ドキュメンテーションの観点

ドキュメント全般（README、docs/、Design Doc、プランファイル等）に共通する品質観点集。

## 正確性（コードとの整合）

ドキュメントの主張が実際のコードと一致するか。

- 記載されたクラス名・関数名・型名が実際に存在し、export されているか
- 記載されたメソッドシグネチャ（引数の型・数・順序、戻り値の型）が実際と一致するか
- 記載されたデフォルト値が実装と一致するか
- 記載された動作説明が実装と一致するか
- 記載されたファイルパス・ディレクトリ構造が実際と一致するか
- バージョン番号や依存関係の記載が package.json と一致するか

## コード例の正しさ

- import パスが正しいか（`exports` フィールドで公開されたパスか）
- 使用している API が実在するか（存在しないメソッド・プロパティの参照がないか）
- 引数の型・数が正しいか
- コード例同士で矛盾がないか（同じ操作を別の方法で書いている等）
- コード例が最新の API を反映しているか（リファクタリング後に古い API が残っていないか）

## 網羅性（利用者向けドキュメント）

- **セットアップ**: インストール手順、前提条件、動作環境
- **基本的な使い方**: 最小限の動作例（コピペで動く）
- **公開 API**: export されたクラス・関数・型の説明
- **設定・カスタマイズ**: オプション、デフォルト値、カスタマイズ方法
- **制約・既知の問題**: 利用者が知るべき制限事項

## 構成・導線

- 読む順序が自然か（概要 → セットアップ → 基本使用 → 詳細 → リファレンス）
- 見出しの粒度が適切か（深すぎず浅すぎず）
- ドキュメント間のリンク・参照が正しいか（リンク切れがないか）
- 目次がある場合、内容と一致しているか

## 読みやすさ

- 対象読者に合った説明レベルか（前提知識の想定が適切か）
- 冗長な説明がないか（同じことを繰り返していないか）
- 専門用語に説明が必要な箇所はないか
- 表・箇条書き・コード例の使い分けが適切か

## 一貫性

- 用語が統一されているか（同じ概念に複数の名前を使っていないか）
- 表記が統一されているか（大文字 / 小文字、ハイフン / キャメル等）
- コード例のスタイルが統一されているか
- 文体が統一されているか

## 図表

- シーケンス図・状態遷移図・フローチャートは Mermaid 形式を優先
- 図で表現した方が分かりやすい箇所をテキストで書いていないか

## 信頼度マーク（✅ / ⚠️ / ❓）

要件・設計項目に、情報源の確かさを示すマークを付ける。

| マーク | 意味 | 条件 |
|---|---|---|
| ✅ | 確認済み | ユーザー発言・コード・ドキュメントから直接確認した事実 |
| ⚠️ | 推測 | 確認済み情報からの妥当な推測 |
| ❓ | 仮定 | 情報源がない仮定。実装前にユーザー確認が必要 |

**例**

```markdown
## 要件
- ✅ 認証は既存の JWT ミドルウェアを使用（auth.go:L42 で確認）
- ⚠️ トークンの有効期限は 24 時間（現在の設定から推測）
- ❓ リフレッシュトークンの要否は未確認
```

**使いどころ**

- Design Doc の要件・設計・実装詳細セクション
- プランファイルの前提条件・不足情報
- レビュー時の指摘根拠

**不要な場所**

- 背景・目的、スコープ、関連コード参照

**注意**

- ❓ が多い項目はヒアリングで解消できないか検討する

## 設計ドキュメントに実装コードを書きすぎない

Design Doc の目的は「何をどう作るか」の設計判断を伝えること。実装コードを書くことではない。コードは設計判断を補足する最小限の断片のみ許可する。

### 基本チェック

- コードブロック合計行数がドキュメント全体の 1/3 を超えていないか
- 各コードブロックを削除しても設計意図が伝わるか（伝わるなら削除）
- クラスの完全実装、関数本体、初期化処理が含まれていないか
- 設計を全部コードで表現しようとしていないか（テキストや図表で十分な箇所はないか）
- 各コード例が 1 つの設計判断に焦点を当てているか（複数の関心事が混在していないか）

### 書いてよい / 書いてはいけないコード

- **書いてよい**: 型定義・インターフェース、データ構造、非自明な分岐ロジック（数行）
- **書いてはいけない**: クラスの完全実装、関数の本体、初期化処理、ボイラープレート、ユーティリティ関数

**判断基準**: そのコードブロックを削除しても設計意図が伝わるなら、そのコードは不要。

**定量目安**: コードブロックの合計行数がドキュメント全体の 1/3 を超えたら書きすぎ。

### Bad 例（設計ドキュメントに不適切）

```typescript
// ❌ クラスの完全実装を書いている
class MovingState implements State<GameData> {
  name() { return "Moving"; }
  entry(machine: EntryMachine<GameData>, event: object): void {
    const data = machine.value();
    if (event instanceof MoveEvent) {
      data.moveDirection = event.direction;
      data.facing = event.direction;
    }
    const tick = (m: AfterFuncMachine<GameData>): void => {
      const d = m.value();
      const dx = d.moveDirection === "right" ? SPEED : -SPEED;
      d.playerX = Math.max(0, Math.min(d.playerX + dx, d.sceneWidthPx - W));
      updateCamera(d);
      checkSceneTransition(m, d);
      m.afterFunc(d.dispatcher, TICK_MS, tick);
    };
    machine.afterFunc(data.dispatcher, TICK_MS, tick);
  }
}
```

### Good 例（設計判断のみ伝える）

> Moving state は afterFunc チェーン（16ms 間隔）でピクセル単位移動を実現する。
> self-transition で前回チェーンが自動キャンセルされるため、方向変更・停止が安全。

```typescript
// 移動 tick 内でシーン遷移ゾーンを検出 → StopMoveEvent + EnterTownEvent
machine.afterFunc(dispatcher, TICK_MS, tick);
```

### その他の注意

- 1 つのコード例 = 1 つの設計判断（複数の関心事を混ぜない）
- 自明な処理は `// ...validation...` のように省略コメントで飛ばす
- コードよりテキスト・表・Mermaid 図が適切な場合はそちらを使う
