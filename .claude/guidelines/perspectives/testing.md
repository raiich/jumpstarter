# テストの観点

テストケース設計・テストコード作成・テストレビュー共通の観点集。

## 網羅性

- 機能要件からの導出漏れがないか（design.md の要件 → テストケースの対応）
- 正常系・異常系・エッジケースのバランス
- 重要なユーザーシナリオがカバーされているか
- エクスポートされた関数 / メソッドにテストが存在するか

## テスト品質

- **振る舞いをテストしているか**: 「ユーザーの操作 → 結果」を検証している。定数値・配列サイズ・enum 数・プロパティ値のアサーションはテストではない
- **バグ検知器か、変更検知器か**: プロダクションコードの正しいリファクタリングでテストが壊れるなら、そのテストは実装に結合している
- **定数の再記述になっていないか**: プロダクションコードの値を変更したとき、テストも機械的に同じ値に書き換えるだけなら、そのテストに価値はない
- **テストの起点は機能要件か**: design.md の要件セクション（「〜したら〜になる」）から導出している。実装詳細（型定義、モジュール構造）から導出していない
- **public API を通してテストしているか**: 内部関数・内部状態を直接テストしない。モジュールの利用者視点で書く
- **テストが使用例のドキュメントになっているか**: テストを読めばそのモジュールの使い方がわかる
- **モックが複雑になっていないか**: モックが膨らむのはテスト対象の責務が大きすぎるサイン。テスト設計かプロダクション設計を見直す

## False positive 防止

- ループ内 assertion の対象が空でないことが保証されているか
- 条件分岐で assertion がスキップされるパターンがないか
- 出力から検証値を作る循環ロジックがないか

## Assertion の妥当性

- 期待値が独立した根拠から導出されているか
- 許容幅（tolerance）が適切か（広すぎてテストが意味をなしていないか）
- 上限・下限の片方しかチェックしていないケースがないか

## 引数の一致

- 関数呼び出しが全必須引数を渡しているか
- TypeScript の型チェックが通っても、ランタイムで undefined になる引数がないか

## テスト構造

- 各テストが独立しているか（共有状態、実行順序への依存がないか）
- Arrange-Act-Assert が明確に分離しているか
- モックのセットアップ / クリーンアップが適切か
- テスト名が振る舞いを表しているか
- DAMP > DRY: ヘルパーで隠しすぎず、各テスト内で何をテストしているかが読み取れることを優先

## 既存テスト基盤の流用

テストケース設計・実装前に、既存のテスト用インフラを調査して再利用する。

**調査対象**
- テスト対象に関連するモック・スタブ・テストヘルパー（`test/`, `mock/` 配下など）
- setUp / tearDown ユーティリティ
- fixtures やテストデータ

**原則**
- 再利用可能なものがあれば優先して活用する
- プロダクションコードのファクトリ・ヘルパーを再利用する（test-helpers に再実装しない）
- テスト固有コード（時間制御・Mock）のみを新規ヘルパーに置く
- 新たに必要な場合（例: 遅延制御や実行順序の観測）は理由を明示する

**判定方法**: テストファイルの import を確認。プロダクションコードと同じ処理が test-helpers に書かれていないか Grep で検索。
