# 一括修正の原則

コードやドキュメントの修正時に、場当たり的・段階的な修正を避け、一度で完結させる原則。

## 基本原則

- **修正方針を決めてから修正する** — 場当たり的に修正しない
- **影響範囲を確認してから修正する** — 関連箇所を全て把握する
- **同様の問題を一度に修正する** — 段階的な修正は避ける

## パターン検索（同類問題の網羅）

修正時、同じ問題パターンが他にもないか必ず検索する。

### 手順

1. **パターン検索**: Grep で同じ問題が他にもないか検索
   - 例: エラーハンドリングの修正 → 同じパターンを全て検索
   - 例: 変数名の修正 → 同じ名前の使用箇所を全て検索

2. **関連ファイル確認**: 影響範囲の特定
   - 例: 関数名変更 → 全ての呼び出し箇所を確認
   - 例: 型定義変更 → 使用している全てのファイルを確認

3. **一括修正**: 見つかった全ての箇所を一度に修正

## 実践例

```
修正タスク: "err 変数の未使用を修正"

❌ 悪いアプローチ:
1. 指摘された 1 箇所を修正
2. ユーザーに報告
3. ユーザー: 「他にもないですか？」
4. 再検索して追加修正...

✅ 良いアプローチ:
1. Grep で "err" の未使用パターンを全検索
2. 見つかった全ての箇所を一度に修正
3. ユーザーに報告: 「5 箇所の未使用 err 変数を修正しました」
```

## 目標

- 「他に漏れはないですか？」の質問をゼロにする
- 段階的な修正を避け、一度で完結した修正を実施
