# 成果物完了フロー

スキル内でドキュメント・コード・設定を作成する際の完了フロー。最初のヒアリングと方針承認を経た後は、途中の承認ゲートを置かずに最後まで自律的に進める。

## 原則

- **承認は最初に 1 回だけ**: ヒアリングで要件・方針を揃え、必要に応じて方針全体の承認を取ったら、以降は途中で承認を求めずに最後まで実行する
- **ヒアリングは開始時にまとめて**: 不明確な要件・方針・選定事項は開始時にまとめて聞く（[hearing.md](hearing.md)）
- **Plan mode の ExitPlanMode** はこの「最初の 1 回の承認」に相当する。フェーズ遷移ごとに繰り返し求めない
- **例外**: 破壊的・不可逆な操作（本番データ削除、外部サービスへの送信、ユーザー環境の破壊的変更など）の直前のみ確認する

## フロー

1. **ヒアリング**: 要件・方針・選定事項を開始時に対話的に確認する（[hearing.md](hearing.md)）
2. **方針承認（必要な場合のみ）**: Plan mode などでスコープ全体の方針をユーザーに承認してもらう。以降は途中承認なし
3. **自律実行**: 成果物（ドキュメント・コード・設定）を最後まで作成する
4. **自己レビュー**: 作成内容を自分で見直す。観点は [../perspectives/design-and-coding.md](../perspectives/design-and-coding.md) / [../perspectives/testing.md](../perspectives/testing.md) / [../perspectives/documentation.md](../perspectives/documentation.md) を参照
5. **ユーザーへの報告**: 成果物の要点・リスク・残課題を簡潔に提示する（承認を求めるのではなく、何を作ったか・次に何が必要かを伝える）

## ❓マーク付き仮定項目

途中で確認が必要な仮定は、原則として開始時のヒアリングで解消する。やむを得ず後回しになった `[❓要確認]` は、対応するタスク着手前に一括で確認する（承認ゲートにはしない）。

## 適用場面

- 各種ドキュメント（Design Doc、テストケース、OKR 等）作成後
- 設定変更後（rules、skills、settings 等）
- 実装完了後
