# 原因分析プロセス

問題（テスト失敗、バグ、予期せぬ挙動）への対処で、場当たり的な修正を避けるためのプロセス。

## プロセス

### 1. 直接原因の特定

エラーメッセージ、スタックトレース、失敗した assertion から直接の原因を特定。

### 2. 上流の原因を探る

- 直接原因のさらに上流を探る（なぜそうなったか？）
- 症状の出た場所が本当の原因か確認
- 同じ問題が他の場所でも起きていないか検索（[fix-in-one-pass.md](fix-in-one-pass.md) のパターン検索を参照）

### 3. 修正候補を複数検討

- この修正は場当たり的でないか？根本原因に対処しているか？
- 他のテストや機能に影響しないか？
- 設計レベルの見直しが必要ではないか？

### 4. 原因の分類と対応

| 分類 | 対応 |
|------|------|
| 実装のバグ | 原因箇所（症状の箇所とは限らない）を修正し再テスト |
| テスト設計のミス | テストケース設計を見直し、tests.md を更新 |
| 設計の問題 | Design Doc に立ち戻り、設計を見直してからやり直す |
| 判断不可 | ユーザーにヒアリング |

## エスカレーション

- **修正が 2 回失敗したら**: 同じアプローチを繰り返さず、別の原因仮説やアプローチを検討する
- **3 回失敗したら**: ユーザーに相談する
