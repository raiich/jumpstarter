# 簡潔さ

## 文章の簡潔さ

❌ 冗長:

```
コンパイラは非常に賢く、使用されていない変数を自動的に削除する最適化を行います。
これは通常は望ましいですが、ベンチマークでは問題となります。
なぜなら測定したい処理が削除されてしまうからです。
この問題を解決するために、グローバル変数への代入を使用します。
```

✅ 簡潔:

```
ベンチマーク対象の処理が最適化で削除されないよう、結果をグローバル変数に代入します。
```

指針: 既知の前置き（「コンパイラは賢く」等）と因果説明（「なぜなら…」「この問題を解決するために…」等）は省き、短くまとめる。

## コードの簡潔さ

- 最小限の変更で目的を達成する
- 早すぎる抽象化を避ける（1 回しか使わない処理の共通化など）
- 仮説的な将来の要件のために設計しない（不要なオプション追加など）
- コメントで処理フローを記述するなら自然言語か疑似コードで、簡潔さ最優先。実コードを再現しない

## ドキュメントとコードの重複

記述は対象に最も近い 1 つの記述先に置く。同じ内容を複数の記述先に書かない。
重複は、同一ファイル内に限らずドキュメント↔コードやコメント間（package doc ↔ 型/メソッドのコメント、別ファイルのコメント）でも起きる。

| 記述先 | 書く内容 |
|---|---|
| ドキュメントファイル | コード横断（複数パッケージ・複数言語）/ 図表・利用手順・導入 / どのコード単位にも属さない横断的 why・代替案 |
| package doc | パッケージ全体の役割・パッケージ固有の設計判断 / 代替案 |
| 型・関数の doc コメント | その API の契約・意図（前提・所有権・エラー方針） |
| inline コメント | 局所の非自明な理由 |

- **重複のサイン**: ドキュメントで「詳細は X」「mechanism はこう」と語る / doc コメントの重複 / 別ファイルに同様のコメントがある
- **解消**: 対象に最も近い記述先（package doc / doc コメント / inline）へ、why は判断の主体に最も近い記述先へ寄せ、重複を削る（内容で監査）

## 陳腐化しやすい表現

時間や実装の進行で実態とズレる表現を、中立・恒常的な記述へ直す。ドキュメントだけでなくコード（コメント・godoc・package doc）も対象。

判定の核心: その記述の「置き場」が恒久的な契約（godoc・package doc・README の構造記述）なら、実装状態・将来参照を持ち込まない。状態追跡が役割の場所（status・マーカー）なら維持する。

### パターンと言い換え

- **現在の実装状態への依存**: 実装済 / 未実装 / サポート組合せ・「のみ対応」・「第 N 単位限定」の列挙。実装が進むと誤りになる。
  - ❌ `未実装の組合せ（relay × udp 等）はエラー扱い`
  - ✅ `上記以外の組合せはエラー扱い`（維持される表・default 挙動として書く）
- **将来・フェーズ参照**: 「将来 X」「Relay 段階で再導入」「後で」。実現で陳腐化。
  - ❌ `（将来フラグ駆動に拡張する場合もこの struct を経由）` → 削除（仮説的な将来要件は書かない）
  - ❌ `Relay 段階で再導入される際に再評価します` → ✅ `設定可能な X を追加する際は再評価する`（行動指示は残し、フェーズ名を外す）
- **歴史的経緯・変更文脈**: 「以前は」「着手前合意を求めた論点への回答」。before・変更理由は成果物に持ち込まない。
- **相対時間・現行名指し**: 「現在」「今」「新しい書き方」「現行ブランチ `foo`」。中立な恒常表現へ。
- **特定バージョン・外部ツールの現況**: 最低要件（「Go 1.24 以上」）は可。流動的な状況説明（「X はメンテ縮小」等）は選定理由として近傍に限定。

### 意図的な状態管理は対象外

進行状態を設計上追跡する記述は陳腐化リークではなく維持する: status ドキュメント、進捗 / 確信度マーカー（✅ / ⚙️ / ❓）、sketch 段階の注記（「コードが panic する」等の利用者向け警告）。
