# 陳腐化しやすい表現

状態・将来・時間に触れる記述を書くとき、中立・恒常的に直すときに参照する。

時間や実装の進行で実態とズレる表現を、中立・恒常的な記述へ直す。ドキュメントだけでなくコード（コメント・godoc・package doc）も対象。

判定の核心: その記述の「置き場」が恒久的な契約（godoc・package doc・README の構造記述）なら、実装状態・将来参照を持ち込まない。状態追跡が役割の場所（status・マーカー）なら維持する。

## パターンと言い換え

- **現在の実装状態への依存**: 実装済 / 未実装 / サポート組合せ・「のみ対応」・「第 N 単位限定」の列挙。実装が進むと誤りになる。
  - ❌ `未実装の組合せ（relay × udp 等）はエラー扱い`
  - ✅ `上記以外の組合せはエラー扱い`（維持される表・default 挙動として書く）
- **将来・フェーズ参照**: 「将来 X」「Relay 段階で再導入」「後で」。実現で陳腐化。
  - ❌ `（将来フラグ駆動に拡張する場合もこの struct を経由）` → 削除（仮説的な将来要件は書かない）
  - ❌ `Relay 段階で再導入される際に再評価します` → ✅ `設定可能な X を追加する際は再評価する`（行動指示は残し、フェーズ名を外す）
- **歴史的経緯・変更文脈**: 「以前は」「着手前合意を求めた論点への回答」。before・変更理由は成果物に持ち込まない。
- **相対時間・現行名指し**: 「現在」「今」「新しい書き方」「現行ブランチ `foo`」。中立な恒常表現へ。
- **特定バージョン・外部ツールの現況**: 最低要件（「Go 1.24 以上」）は可。流動的な状況説明（「X はメンテ縮小」等）は選定理由として近傍に限定。

## 意図的な状態管理は対象外

進行状態を設計上追跡する記述は陳腐化リークではなく維持する: status ドキュメント、進捗 / 確信度マーカー（✅ / ⚙️ / ❓）、sketch 段階の注記（「コードが panic する」等の利用者向け警告）。
