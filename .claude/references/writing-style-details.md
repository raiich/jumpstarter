# 記述スタイル詳細

記述スタイルの具体例・語彙の判断材料を提供する。

## 簡潔さ

### 文章の簡潔さ

❌ 冗長:

```
コンパイラは非常に賢く、使用されていない変数を自動的に削除する最適化を行います。
これは通常は望ましいですが、ベンチマークでは問題となります。
なぜなら測定したい処理が削除されてしまうからです。
この問題を解決するために、グローバル変数への代入を使用します。
```

✅ 簡潔:

```
ベンチマーク対象の処理が最適化で削除されないよう、結果をグローバル変数に代入します。
```

指針: 既知の前置き（「コンパイラは賢く」等）と因果説明（「なぜなら…」「この問題を解決するために…」等）は省き、短くまとめる。

### コードの簡潔さ

- 最小限の変更で目的を達成する
- 早すぎる抽象化を避ける（1 回しか使わない処理の共通化など）
- 仮説的な将来の要件のために設計しない（不要なオプション追加など）
- コメントで処理フローを記述するなら自然言語か疑似コードで、簡潔さ最優先。実コードを再現しない

## 単語選択

特定の語を使う/避ける判断、または強い語彙を中立化する判断をするときに参照する。

### 語彙のトーン: 強い語彙の中立化

強い語彙（評価・感情を含意する語）は事実ベースの中立表現に置き換える。

#### 避ける語と置換例

| 強い語 | 置換例 |
|---|---|
| 毀損する | 品質が下がる / 成立しなくなる |
| 破綻する | 成り立たない |
| 破壊する | 対応関係が崩れる |
| 台無し | 成り立たなくなる |
| 崩壊 | 整合性が失われる |

#### 判定の目安

- 同じ事実を、感情・評価を含まない言い方で書けるなら書き換える
- 文の中心に「実害」「観測される現象」を据える

#### 例外

外部仕様・既存ラベル（例: `Critical` / `Major` / `Minor`）はそのまま用いる。中立化の対象は地の文の語選び。

### 標準用語の選択

既存の手法・フレームワークを参照するときは、標準的な用語に合わせる。意図的に意訳する場合のみ、本文または近傍にその旨を明示する。

出典の優先順位（高→低）: 公式サイト・創始者の原典 → 標準化団体の規格（ISO、IEEE 等） → 業界の通例・教科書

### 慣用語を優先する（無理に漢字へ訳さない）

定着した英語・カタカナ語を無理に漢字へ訳さない（赤緑、長尾等）。漢字が慣用ならそのまま使う（名前解決、転置インデックス等）。

### 「技術」とツール/サービスの区別

「技術」は問題を解決するための体系的な手法・技法の意味で使い、特定のソフトウェア・サービス・ツールとは区別する。

#### 分類

| 分類 | 例 |
|------|-----|
| 技術 | パケットルーティング、名前解決、動的計画法、転置インデックス、コンテナ隔離 |
| ソフトウェア/ツール | Docker, Rails, AWS, Hadoop, Git |
| 仕様/プロトコル | TCP/IP, HTTP, SIP |

#### 表現例

- ❌ 「Docker という技術」「React 技術を導入」
- ✅ 「コンテナ技術（実装: Docker）」「コンポーネントベースの UI 構築手法（React が採用）」
