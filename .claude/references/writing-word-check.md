# 単語選択（語彙の方針とチェックリスト）

特定の語を使う/避ける判断、または強い語彙を中立化する判断をするときに参照する。

## 語彙のトーン: 強い語彙の中立化

強い語彙（評価・感情を含意する語）は事実ベースの中立表現に置き換える。対象語と置換例は末尾の[チェックリスト](#チェックリスト)。

### 判定の目安

- 同じ事実を、感情・評価を含まない言い方で書けるなら書き換える
- 文の中心に「実害」「観測される現象」を据える

### 例外

外部仕様・既存ラベル（例: `Critical` / `Major` / `Minor`）はそのまま用いる。中立化の対象は地の文の語選び。

## 標準用語の選択

既存の手法・フレームワークを参照するときは、標準的な用語に合わせる。意図的に意訳する場合のみ、本文または近傍にその旨を明示する。

出典の優先順位（高→低）: 公式サイト・創始者の原典 → 標準化団体の規格（ISO、IEEE 等） → 業界の通例・教科書

## 慣用語を優先する（無理に漢字へ訳さない）

定着した英語・カタカナ語を無理に漢字へ訳さない（赤緑、長尾等）。漢字が慣用ならそのまま使う（名前解決、転置インデックス等）。

## 「技術」とツール/サービスの区別

「技術」は問題を解決するための体系的な手法・技法の意味で使い、特定のソフトウェア・サービス・ツールとは区別する。

### 分類

| 分類 | 例 |
|------|-----|
| 技術 | パケットルーティング、名前解決、動的計画法、転置インデックス、コンテナ隔離 |
| ソフトウェア/ツール | Docker, Rails, AWS, Hadoop, Git |
| 仕様/プロトコル | TCP/IP, HTTP, SIP |

### 表現例

- ❌ 「Docker という技術」「React 技術を導入」
- ✅ 「コンテナ技術（実装: Docker）」「コンポーネントベースの UI 構築手法（React が採用）」

### 参考

- [「技術」という言葉について](https://kuenishi.hatenadiary.jp/entry/2015/04/17/001713)

## チェックリスト

`.claude/hooks/check-wording.sh` が PostToolUse で照合する要検討語と置換案。各行 `パターン ::: 指摘文`。`exact` は固定文字列一致、`regex` は正規表現。語幹で登録する（活用形を拾う）。

```exact
毀損 ::: 品質が下がる／成立しなくなる 等
破綻 ::: 成り立たない
破壊 ::: 対応関係が崩れる
台無し ::: 成り立たなくなる
崩壊 ::: 整合性が失われる
指摘を受けて ::: 変更文脈は持ち込まない（成果物は結果状態のみ）
変更前は ::: before/変更理由を残さない
修正前は ::: before/変更理由を残さない
以前は ::: 相対時間は中立な恒常表現へ
現行 ::: 現行〜の名指しを避け恒常表現へ
xxx ::: サンプル値は foo / 192.0.2.x / 2001:db8::/32 / example.com 等
XXX ::: サンプル値は foo / 192.0.2.x / 2001:db8::/32 / example.com 等
```

```regex
技術 ::: 「技術」はツールと区別（例: コンテナ技術（実装: Docker））
```
