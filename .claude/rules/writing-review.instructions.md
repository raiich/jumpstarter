# セルフレビュー

成果物（応答・ドキュメント・コード）に対し、下記サインをスキャンして当たり判定する。応答送信前、およびファイルの記述・修正後に実行する。サインが出たら「直す」へ。

サインは違反を拾う入口であり、許可リストではない。字句一致でなく意味で判定する — 字句を変えた同等違反や一覧外の表現も対象、逆に文脈上正当なもの（エラー原因の `なぜなら`、承認質問の `〜しますか`）は残す。サイン不在は合格を意味しない。直し方に置換語・記述先・判定基準が要るときは、節末のリファレンスを開く。

方針そのものは [書き方の方針](writing-style.instructions.md)。

## 先回り・冗長

[簡潔さ](../references/writing-conciseness.md) に従う。

- 既知の前置き（「X は十分賢く、〜」等）→ 前置きを削る
- ユーザー入力と照合 → 要求外の先回り説明・プロンプトの背景説明の紛れ込みを削る
- `なぜなら` / `〜のため` → 因果を削る。1 セクションに 3 回以上出たら冗長さのサイン。簡潔にできないか試す
- 1 段落に 4 文以上、または並列・列挙できる論点が混在 → 冗長さのサイン。不要文を削り、残りが列挙なら箇条書き、地続きの論理なら圧縮する

## 陳腐化・変更文脈・やり取りの持ち込み

- `現在` / `今` / `以前は` / `将来` / `新しい` / `現行` ＋ ブランチ名・バージョン直書き → 恒常表現へ（[陳腐化](../references/writing-staleness.md)）
- 実装状態の列挙（`未実装の組合せ` / `〜のみ対応`）→ 維持される表・default 挙動として書く
- `〜への対応` / `変更前は` / `指摘を受けて` / before → 成果物から削る
- 「（メイン）/（サブ）」等のやり取り用の語 → 並び順・本文で意図を表す

## 重複

- `詳細は X` と書いた上で同内容を両所に置く / ドキュメント↔コメントで同じ事実 → 対象に最も近い 1 箇所へ寄せる（[重複と記述先](../references/writing-redundancy.md)）

## 強調・記法

- 1 段落に太字・絵文字が複数、または見出し直下の即強調 → 注意喚起以外は外す
- 表が 3 列以上 → 箇条書きで代替できるか検討
- サンプル値にスラング・実在リソース → `foo` / `192.0.2.x` / `example.com` 等へ
- 文体の混在（です・ます調 ↔ だ・である調）→ 記述先の文体へ統一

## 語彙

固定シグネチャはフックが検出する。本節で見るのはフックが拾えない残差。

- 言い換え後に事実が中立に伝わるか（置換で意味がぼけていないか）
- 技術とツール／サービスの意味的区別（シグネチャに載らない言い回し）

## コード

- 1 回しか呼ばれない共通化（抽出先が単一呼び出し元）→ inline 化を検討
- 使われていないオプション・フラグ・引数の追加 → 削る
- 要求にない汎化（interface 化・将来用パラメータ）→ 最小実装へ
- diff が要求より広い（無関係な整形・リネーム）→ 分離

## 定性確認（サインで拾えない残差）

機械的なサインでは判定できない。書き手の同一パスでは盲点が残るため、独立した見直し（別パス・別エージェント）に回す。

- 周辺の記述と文体・粒度がずれていないか
- 各記述を「削って伝わるか」で見直し、冗長・重複を落とす
