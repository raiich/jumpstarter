# 記述スタイル

関心ごとに **方針**（どう書くか）と **検証と修正**（検証方法＋修正方針）を定める。応答送信前・ファイルの記述/修正後に当てる。検証は文字列一致でなく意味で行い（文脈上正当なもの——エラー原因の `なぜなら`、承認質問の `〜しますか`——は残す）、該当がなくても合格とは限らず、機械的に拾えない品質は末尾の[残差](#残差)へ。

`.claude/` / `.github/` 配下のルール・スキル定義ファイル自体も本ガイドラインの対象。

## 簡潔さ

**方針**

- 簡潔に必要十分な情報を出力。先回り・自明な説明（既知の前置き・因果説明）を避ける
- タイムライン・工数見積もりは書かない

**検証と修正**

- 削っても依頼への回答が変わらない前置き（読み手の既知事実。「X は十分賢く〜」等）→ 削る
- 依頼と照合し、聞かれていない先回り説明 → 削る
- `なぜなら` / `〜のため` → 因果節を削る（エラー原因等、文脈上正当なものは残す。1 セクション 3 回以上は特に圧縮）
- 1 段落 4 文以上、または並列・列挙できる論点の混在 → 不要文を削り、列挙は箇条書き・地続きは圧縮

例: 「コンパイラは賢く未使用変数を削除する。これは通常望ましいが…なぜなら…この問題を解決するため…」→「ベンチマーク対象が最適化で削除されないよう、結果をグローバル変数に代入する」（既知の前置きと因果を省く）

## 陳腐化・変更文脈

**方針**

- 状態・将来・時間に依存せず中立・恒常的に書く。置き場が恒久的な契約（doc コメント・package doc・README の構造記述）なら、実装状態・将来参照を持ち込まない
- 成果物は結果状態のみを記述。変更文脈（理由・指示・before）とやり取り用の語を持ち込まない。意図（優先度・強調・スコープ）は構成・順序・本文で表す

**検証と修正**

- 相対時間・現行名指し（`現在` / `今` / `以前は` / `将来` / `新しい` / `現行`）＋ ブランチ名・バージョン直書き → 中立な恒常表現へ
- 実装状態の列挙（`未実装の組合せ` / `〜のみ対応`）→ 維持される表・default 挙動として書く（例: `未実装の組合せ（…）はエラー` → `上記以外の組合せはエラー`）
- 将来・フェーズ参照（`将来 X` / `フェーズ B で再導入` / `後で`）→ 仮説的な将来要件は削除、行動指示はフェーズ名を外して残す（例: `フェーズ B で再導入される際に再評価` → `設定可能な X を追加する際は再評価する`）
- 出力を依頼と照合 → 残った依頼の反復・言い換え・変更文脈（`〜への対応` / `変更前は` / `指摘を受けて` / before）・やり取り用の語（「（メイン）/（サブ）」等）を削る。意図は並び順・本文で表す

対象外（維持する）: 状態追跡が役割の記述 — status ドキュメント、進捗/確信度マーカー（✅ / ⚙️ / ❓）、sketch 段階の利用者向け警告。最低要件（`Go 1.24 以上`）は可。流動的な状況説明（`X はメンテ縮小` 等）は選定理由として近傍に限定。

## 重複・記述先

**方針**

- 重複をなくし、記述を最適な 1 箇所へ置く。ドキュメントよりコード（コメント含む）へ寄せ、ドキュメントには図・横断的内容を置く
- why は判断の主体に最も近い記述先へ寄せる

記述先:

- ドキュメントファイル: コード横断（複数パッケージ・言語）/ 図表・利用手順・導入 / どのコード単位にも属さない横断的な why・代替案
- package doc: パッケージ全体の役割・固有の設計判断 / 代替案
- 型・関数の doc コメント: その API の契約・意図（前提・所有権・エラー方針）
- inline コメント: 局所の非自明な理由

**検証と修正**

- 出力を比較し、同内容が複数箇所にあれば最も近い 1 箇所へ寄せ重複を削る（同一ファイル内・ドキュメント間・コード↔ドキュメント/コメント間。「`詳細は X`」＋本文再掲が典型）

## 記法・強調

**方針**

- ファイル名: lowercase-with-hyphens (kebab-case)。例外: README.md, CLAUDE.md 等の慣習的な大文字
- 図は mermaid を優先
- 文体: です・ます調（一般のドキュメント・応答）。`.claude/` / `.github/` 配下はだ・である調
- 絵文字・強調は特殊な注意喚起（構造・重要性の明示、視認性向上）に限る
- サンプル値はスラング・実在リソースを避ける
- 表は列数を抑える。箇条書きでの代替も検討
- コードブロックはファイル・Before/After 等で分ける
- 段落を分け、価値・手順・詳細・注意点を関心ごとに記載

**検証と修正**

- 1 段落に太字・絵文字が複数、または見出し直下の即強調 → 注意喚起（構造・重要性の明示）でなければ外す
- 表が 3 列以上 → 箇条書きで代替できるか検討
- 出力中の仮名・IP・URL・メールが規定値（`foo` / `192.0.2.x` / `2001:db8::/32` / `example.com` / `user@example.com`）でなければ置換
- です・ます調 ↔ だ・である調 の混在 → 記述先の文体へ統一

## 用語・語彙

語彙の方針（強い語彙の中立化・標準用語・手法とツールの区別）は [単語チェックリスト](../references/writing-word-check.md)。固定シグネチャはフックが検出する。フックが拾えない残差を見る:

- 言い換え後に事実が中立に伝わるか（置換で意味がぼけていないか）
- 手法・技法とソフトウェア/サービスを混同した言い回し（固定シグネチャに載らない形）

## コード

**方針**

- 最小変更で達成。早すぎる抽象化（1 回しか使わない共通化）・仮説的な将来要件の設計を避ける
- コメントで処理フローを書くなら自然言語/疑似コードで簡潔に。実コードを再現しない

**検証と修正**

- 共通化の呼び出し元が 1 回のみ → inline 化を検討
- オプション・フラグ・引数が依頼にない or 未使用 → 削る
- interface 化・将来用パラメータが依頼にない → 最小実装へ
- diff が依頼の範囲より広い（無関係な整形・リネーム）→ 分離

## 残差

機械的な検証では判定できない品質。書き手の同一パスでは盲点が残るため、独立した見直し（別パス・別エージェント/セッション）に回す。

- 周辺の記述と一貫しているか（粒度・トーンのずれ。追記・変更箇所も同様）
- 各記述を「削って伝わるか」で見直し、冗長・重複を落とす
