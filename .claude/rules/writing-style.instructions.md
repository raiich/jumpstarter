# 記述スタイルガイドライン

ドキュメント・コード（コメント・doc コメントを含む）・対話応答のすべてに適用する。`.claude/` / `.github/` 等の配下のルール・スキル定義ファイル自体も対象。

## 簡潔さ

**最優先**: 「簡潔かつ必要十分な出力」でない場合、内容が正しくても不十分。

- 必要十分な情報のみ。短い文章、明確な用語、箇条書きの活用
- 過度な説明を避ける（読者が既知、簡単に調べられる情報、聞かれていない先回り、自明なコードの説明）
- 陳腐化する前提に依存しない: 歴史的経緯・特定バージョン・特殊な挙動などに依存しない中立的な書き方をする（古くなっても検知されず誤誘導になる）
- 書いたら削る: 各文・節・修飾語を「削っても伝わるか」で見直す。既知の情報・既存の項目との重複をなくす。半分に削れないか問う
- タイムライン・工数見積もりは不要

判定に迷ったら [記述スタイル詳細の「簡潔さ」](../references/writing-style-details.md#簡潔さ) を参照する。

## 記法

- **ファイル名**: lowercase-with-hyphens (kebab-case)。
  例外: README.md, CLAUDE.md 等の慣習的な大文字ファイル
- **図表**: 図は mermaid を優先
- **文体**: 対象で使い分ける
  - **だ・である調**: `.claude/` / `.github/` 配下のルール・スキル定義
  - **です・ます調**: それ以外（`README.md`、ユーザーへの対話応答、doc コメント / API ドキュメント等）
- **絵文字**: 見逃しやすい場所（列挙箇所等）、視認性の向上が必要な箇所に使う。過剰な使用は避ける
- **強調**: 太線（`**...**`）は、構造や重要性を明示するときに使う。過剰な使用は避ける
- **サンプル値**: 英語も含めたスラング連想や実在リソース（IP・ドメイン・メール等）を避け、誤アクセス・公開しても問題ない表現にする。
  例: 名前 `foo`、IPv4 `192.0.2.x`、IPv6 `2001:db8::/32`、ドメイン `example.com`
- **表**: 列数を抑える（増えるなら箇条書き等を検討）。セル内は短くし（1～2文）、長い説明は表の外に書く
- **コードブロック**: 1ブロック1対象。異なるファイル・Before/After は別ブロックに分ける
- **パターン整合性**: スタイル・記法は、周辺の関連記述（同ファイル内・近接記述・同種ドキュメント群）と一貫させる。
  - 追記・変更箇所を周囲と同等に扱い、目立たせない（特殊な注意喚起が要る場合のみ強調）

## セルフレビュー

記述・修正後、および対話応答の送信前に必ずセルフレビュー。

- 本ファイル全節の観点でレビュー。簡潔さ（書いたら削る）は最優先。
- 変更の影響を分析し、参照箇所との整合性を確認。
