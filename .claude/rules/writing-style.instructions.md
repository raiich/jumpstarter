# 記述スタイルガイドライン

## 簡潔な記述スタイル

- 必要十分な情報のみ。短い文章、明確な用語、箇条書きの活用
- 前置き、過剰な敬語、不要な確認を避ける
- 過度な説明を避ける（読者が既知、または自分で調べられると想定される情報）
- タイムライン・工数見積もりは明示的に求められた場合のみ
- 自明なコード変更に長い説明を加えない
- ユーザーが聞いていないことを先回りして説明しない
- 不確実な点は正直に伝える

## コードの簡潔さ原則

- 最小限の変更で目的を達成する
- 1回しか使わない処理を抽象化しない
- 現時点で不要なオプション・設定を追加しない
- 類似の7行のコードは、早すぎる抽象化より良い
- 仮説的な将来の要件のために設計しない

## スタイル

- **ファイル名**: lowercase-with-hyphens (kebab-case)。例外: README.md, CLAUDE.md 等の慣習的な大文字ファイル
- **図表**: シーケンス図・状態遷移図・フローチャートは mermaid を優先
- **文体**: ですます調を推奨（技術記事として読みやすい）
- **絵文字**: 目立たせたい場所、見逃しやすい場所、見分けがつきにくい列挙箇所に限定
- **表**: セル内は短い内容のみ（1～2文以内）。長い説明は表の外に書く
- **コードブロック**: 1ブロック1対象。異なるファイル・Before/Afterは別ブロックに分ける
- コード例で十分説明できることを長文で書かない

## セルフレビュー

- 各セクションで自問: 削除しても理解できるか？ 既知情報の説明か？ 重複していないか？ 半分に削れないか？
- 批判的に見る: 「良さそう」で済ませず弱点・リスクを探す。手段と目的の整合性を確認する
- 影響分析: 変更した要素を参照している箇所を Grep で検索し、整合性を確認する

## 用語

- ツール名（Docker, React 等）を「技術」と呼ばない
- 背後にある手法・原理を「技術」として記述し、ツールはその実装として言及する
