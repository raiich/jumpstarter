# 書き方の方針

執筆時に「どう書くか」を決める方針。出力後の検証は [セルフレビュー](writing-review.instructions.md) で行う。
`.claude/` / `.github/` 配下のルール・スキル定義ファイル自体も本ガイドラインの対象（文体はだ・である調）。

## 簡潔さ

- 簡潔に必要十分な情報を出力。聞かれていない先回り、自明な説明（既知の前置き・因果説明）を避ける
- 陳腐化回避: 状態・将来・時間に依存せず、中立・恒常的に書く
- 重複をなくし、記述を対象に最も近い 1 箇所へ置く
- ドキュメントよりコード（コメント含む）へ寄せ、ドキュメントには図・横断的な内容を置く
- コードは最小変更で達成。早すぎる抽象化・仮説的な将来要件のための設計を避ける
- ユーザー入力の「やり取りのための語」を出力に持ち込まない。意図（優先度・強調・スコープ）は構成・順序・本文で表す
- 成果物は結果状態のみを記述。変更の文脈（理由・指示・before）を持ち込まない
- 周辺の記述と一貫させる。追記・変更箇所も同様
- 強調は特殊な注意喚起が要る場合のみ
- タイムライン・工数見積もりは不要

詳細: [簡潔さ](../references/writing-conciseness.md) / [重複と記述先](../references/writing-redundancy.md) / [陳腐化](../references/writing-staleness.md)

## 記法

- **ファイル名**: lowercase-with-hyphens (kebab-case)。例外: README.md, CLAUDE.md 等の慣習的な大文字ファイル
- **図表**: 図は mermaid を優先
- **文体**: です・ます調（一般のドキュメント・応答）
- **絵文字** / **強調**: 構造や重要性を明示するとき、視認性の向上が必要な箇所に使う。過剰な使用は避ける
- **サンプル値**: スラングや実在リソースを避ける。例: `foo` / `192.0.2.x` / `2001:db8::/32` / `example.com` / `user@example.com`
- **表**: 列数を抑える。表の代わりに箇条書き等も検討する
- **コードブロック**: ファイル・Before/After などでブロックを分ける
- **関心の分離**: 段落を分けて、価値・手順・詳細・注意点などを記載

## 用語・語彙

語彙の方針（強い語彙の中立化・標準用語）は [単語チェックリスト](../references/writing-word-check.md)。フックが検出する。
