# Ask Before Task

コードベース調査と要件ヒアリングで、タスク（やること・スコープ・方針・前提・仮定）の解釈を 1 つに確定するスキル。

動作環境別の挙動は [environment.md](environment.md) を参照（実行前に一度読み込む）。

## 曖昧さの分類

| 分類 | 何が不明か | 例 |
|---|---|---|
| 対象 | 何に対してか | 「これ直して」「整理して」 |
| スコープ | どこまで・どの粒度か | 「README 用意して」「ヘルパー作って」 |
| 方針 | どう実現するか（可読性 / 性能 / 拡張性 等） | 「リファクタして」 |
| 制約 | 守るべき制約（互換性、ダウンタイム 等） | 「移行して」 |
| 成功基準 | 何が満たされれば完了か | 「使いやすくして」 |

成果物作成系（「〜作って」「〜書いて」「〜追加して」）は対象が明示されていてもスコープ・方針・成功基準が抜けやすい。

## フロー

1. **曖昧さの特定** — 解釈が分かれる箇所を 1〜3 個、上記分類で特定
2. **コードベース調査** — 既知パス/シンボルはファイル読み取り・コード検索、広範な探索は探索エージェントに範囲を指定して並列に。
  「コードを読めば分かること」をヒアリングで聞かない（[references/ask-the-user.md](references/ask-the-user.md) 第一原則）
3. **ヒアリング** — 調査で残った不足情報のみ構造化ヒアリング（1ラウンド最大 4 問）。[references/ask-the-user.md](references/ask-the-user.md) に従う。
  「自律的に進めて」と明示があっても、複数候補ある場合は 1 ラウンドのヒアリングを許容する（承認を取るためではなく方針確定のため）。
  合理的解釈が成立する場合はヒアリングをスキップし、本作業の前提として明示する
4. **解釈の確定** — 下記形式で短く提示してから本作業へ。ここで再度承認は取らない

```markdown
## 着手前の理解

- **やること**: [1〜2 文]
- **スコープ**: 含む / 含まない
- **方針**: アプローチと選択理由
- **前提**: ✅ 調査で確認した既存制約
- **推測**: ⚠️ 確認済み情報からの妥当な推測（あれば）
- **仮定**: ❓ ヒアリング不足で仮定した事項（あれば）
```
