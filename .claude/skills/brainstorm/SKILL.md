# Brainstorm

複数の思考フレームワークを使い、アイデアを多角的に検証するスキルです。
サブエージェントを並列に起動し、最大3ラウンドの反復議論を経て結果を構造化します。

## フロー概要

```
1-4: 準備 → R1: 独立分析（発散） → R2: クロスレビュー（交差） → R3: 収束統合（条件付き） → 結果統合
```

## フロー

### 1. テーマの確認

ユーザーから渡されたテーマを確認: `$ARGUMENTS`

テーマが不明確な場合は AskUserQuestion でヒアリングする。

### 2. エージェント選定

テーマの性質に応じて、起動するサブエージェントを **2〜3個** 選定する。
全てを常に起動する必要はない。

**選定の原則**: 批判的視点と建設的視点の両方を含めること。

| サブエージェント | 視点 | 選定基準 |
|---|---|---|
| `critical-analyst` | 批判 | ほぼ常に有用。反論と失敗予測 |
| `framework-analyst` | 探索 | 問題の構造を理解したい場合。アナロジー・問題パターン・素朴な疑問 |
| `constructive-analyst` | 建設 | 機会発見・強みの拡張。批判だけで終わらせたくない場合 |
| `stakeholder-sim` | 多面 | 複数の利害関係者がいる場合、市場向けプロダクトの場合 |

選定理由は結果報告時にまとめて伝える（ここではユーザー確認を挟まず、そのまま起動に進む）。

### 3. ランダム制約の生成

以下のリストから **2つ** を直感的に選ぶ。前回と同じ組み合わせにならないよう意識する。
テーマに関連しすぎるものより、意外性のある組み合わせを優先する。
リストにない制約を自分で考えても良い。

- 予算が現在の1/10だったら？
- ターゲットが真逆の属性だったら？
- この技術が使えなくなったら？
- 競合が3ヶ月後に同じものを出したら？
- 規模が100倍になったら？
- チームが半分になったら？
- 納期が2倍になったら？
- 法規制が変わったら？
- 主要顧客が離脱したら？
- コア技術の特許が切れたら？

この制約は `stakeholder-sim` を起動する場合に渡す。

### 4. コンテキスト収集（必要な場合）

テーマがコードベースに関連する場合のみ、関連ファイルや既存設計を簡潔に把握する。

**ツール**: Read, Grep, Glob

### 5. Round 1: 独立分析（発散）

選定したサブエージェントを **並列で** 起動する（順次実行しない）。

各サブエージェントには以下を伝える:

```
テーマ: [ユーザーのテーマ]
コンテキスト: [収集した関連情報があれば]
```

`stakeholder-sim` にはランダム制約も追加:
```
ランダム制約: [ステップ3で生成された制約]
```

**ツール**: Agent

### 6. Round 2: クロスレビュー（交差）

R1 で起動した **同じエージェント** を再度 **並列で** 起動する。
各エージェントに、全エージェント（自身含む）の R1 出力を渡し、クロスレビューを指示する。

#### R2 プロンプト構成

```
テーマ: [テーマ]
コンテキスト: [元のコンテキスト]

--- Round 1 の全エージェント分析結果 ---

## [エージェント名A] の分析:
[R1 出力]

## [エージェント名B] の分析:
[R1 出力]

## [エージェント名C] の分析:
[R1 出力]

---

クロスレビュー指示:
他のエージェントの分析を踏まえ、自身の専門的視点から以下を行ってください:
1. 見落とされている点の指摘
2. 他の分析への反論または補強（根拠付き）
3. 複数の分析を組み合わせることで生まれる新たな着想

重要: Round 1 の自身の分析を繰り返さないこと。新しい発見のみを報告すること。
```

#### R1 出力の要約

R1 出力の合計が長大な場合、各エージェントの出力を主要ポイント 5〜7 個程度に要約してから R2 に渡す。

**ツール**: Agent

### 7. Round 3: 判定と収束統合（条件付き）

R2 出力を評価し、以下の **いずれかに該当する場合のみ** 収束統合を実行する。

| トリガー | 判定基準 |
|---|---|
| 重要な矛盾 | R2 エージェント間で同一論点に対し正反対の結論がある |
| 大きな新着想 | R1 になかった実質的に新しいアイデアや視点が R2 で生まれた |
| 未解決トレードオフ | リスク vs 機会、ステークホルダー間の対立が解消されていない |

**該当する場合**: オーケストレータ自身が以下を行う（サブエージェント起動なし）:
- 矛盾の解消または両立条件の明示
- 合意点の特定
- 優先順位付け

**該当しない場合**: スキップしてステップ 8 へ。

### 8. 結果の統合と構造化

全ラウンド（R1 + R2 + R3）の出力を統合し、以下に構造化する。
重複する指摘は統合し、情報量を最大化する。
R2 で深化・補強された内容を優先的に反映する。

**重要**: これは AI による分析です。実際の人間からのフィードバックと組み合わせて使ってください。

#### 出力フォーマット

以下のセクションから、実際にサブエージェントの出力に該当するものだけを含める。
空のセクションは省略する。

```markdown
## ブレスト結果: [テーマ]

### 分析プロセス
- **ラウンド数**: [2 or 3]
- **参加エージェント**: [エージェント名リスト]
- **選定理由**: [簡潔な理由]
- **R3 判定**: [実行した場合はトリガー / スキップした場合は「不要」]

### 未検証の前提リスト
このアイデアが暗黙に前提としていること。検証が必要。
- [ ] 前提1: [内容] — 検証方法: [どうすれば確認できるか]

### リスクリスト（優先度順）
- [ ] リスク1: [内容] (発生確率: 高/中/低, 影響度: 高/中/低)
  - 予防策: [説明]
  - 早期警戒指標: [シグナル]

### 問題構造の分析
この問題が当てはまる既知のパターンと、その既知の解決策。
- パターン: [名前] — 本テーマへの適用: [説明]

### アナロジーからのヒント
他領域の構造的に類似する問題から得られた転用可能なパターン。
- [領域]: [事例] → 転用: [説明]

### 初心者の問い
専門家が見落としがちな素朴な疑問。
- [ ] 問い: [内容] — 重要な理由: [説明]

### TRIZ 発想
発明原理から得られた着想。
- [原理名]: [テーマへの具体的な着想]

### 機会と強み
活かすべきポイント、隠れた機会。根拠付き。
- 機会1: [内容] — 根拠: [説明]

### 検証すべき質問リスト
実際のユーザー・市場・関係者に確認すべき質問。
- [ ] 質問1: [内容] — 確認先: [誰に聞くべきか]

### 発見された代替アプローチ
- 案1: [概要] — 出典: [どの分析から]

### 次のアクション（推奨）
1. [最優先で検証すべきこと]
2. [次に検証すべきこと]
3. [その次に検証すべきこと]
```
