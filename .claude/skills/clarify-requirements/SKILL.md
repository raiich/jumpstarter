# Clarify Requirements

機能開発の前段として、コードベース調査とユーザーヒアリングを通じて要件を明確化するスキルです。

## 前提

- ユーザーは初期段階で全要件を伝えるとは限らない
- コードベースを把握した上で的確な質問をする
- 成果物は `requirements.md` として保存し、`develop-feature` スキルへの入力となる

## 保存先

- **ナレッジベース**: `docs/` 配下にトピックごとのファイル
- **フィーチャー固有ドキュメント**: `docs/features/[名前]/`
  - `requirements.md` - 要件定義（本スキルの主成果物）
- **作業要点**: `.local/claude/learnings.md`（追記式）

## 基本パターン：成果物作成とレビュー

1. **Claude: 成果物作成**
2. **Claude: 自己レビュー** - 必要に応じて修正
3. **ユーザー: レビュー** - 自己レビュー後は必ずユーザーレビューを実施
4. **Claude: 修正** - フィードバックがあれば修正
5. **繰り返し**: ユーザーが承認するまで上記を繰り返す

フローで**基本パターン**と言及された箇所はこのパターンに従います。

## フロー

### [調査・ヒアリングフェーズ] — Plan mode

#### 1. ユーザー: 機能の概要や方針の入力

#### 2. Plan mode に入る

**ツール**: EnterPlanMode

#### 3. コードベース調査

既存コードベースとドキュメント（`docs/` 配下のナレッジベース含む）を調査し、プランファイルに記録。
広範な探索が必要な場合は Task（Explore エージェント）を活用。

**ツール**: Read, Glob, Grep, Task

#### 4. 要件のヒアリング

調査結果と既存ナレッジを踏まえて、不足情報をユーザーにヒアリング。
ヒアリング内容をプランファイルに反映。必要に応じて複数回ヒアリングする。

**ツール**: AskUserQuestion

**プランファイルに記録する内容:**
```markdown
# 調査結果

## 関連する既存機能
- 機能A: path/to/file.go

## 重要なアーキテクチャパターン
- パターン1: 説明

## 参考ドキュメント
- docs/xxx.md

## ヒアリング結果
- 確認した要件・制約
```

#### 5. 自己レビューと Plan mode 終了

プランファイルを自己レビューし、ExitPlanMode でユーザー承認を求める。

**ツール**: ExitPlanMode

### [成果物作成フェーズ] — 通常 mode

#### 6. requirements.md 作成

プランファイルの調査結果を基に `requirements.md` を作成。
作成後は**基本パターン**（自己レビュー → ユーザーレビュー → 修正）に従い、ユーザー承認を得る。

**保存先**: `docs/features/[名前]/requirements.md`

**内容:**
```markdown
# Feature: [名前]

## 背景・目的
- What: 何を作るか
- Why: なぜ必要か

## 要件
- 機能要件
- 非機能要件
- 制約条件

## 関連コード調査結果
- 関連する既存機能とファイルパス
- 重要なアーキテクチャパターン

## 技術方針の候補
- アプローチA: 概要と利点・欠点
- アプローチB: 概要と利点・欠点

## ヒアリング結果サマリ
- 確認した内容
```

**ツール**: Write, Edit, AskUserQuestion

**⛔ ユーザーの承認なしに次へ進まない**

### [完了フェーズ]

#### 7. ナレッジベース更新

調査で得た知見を `docs/` 配下にファイルとして保存・更新。

**対象となる知見:**
- 要件、背景、設計方針、機能とコードの対応
- 検討して採用しなかった代替案とその理由
- 技術的制約・既知の制限事項

※ `.local/claude/learnings.md`（ステップ8）は Claude の作業プロセスに関する学びを記録する場であり、棲み分けること。

**ツール**: Write, Edit

#### 8. 作業要点の保存

ユーザーとのやり取りから得た要点を `.local/claude/learnings.md` に追記：

- **方針**: ユーザーが示した開発方針や意思決定の背景
- **考慮すべき観点**: ユーザーが重視する観点や判断基準
- **進め方のコツ**: タスク進行で効果的だったやり方や段取り
- **注意点**: 避けるべきこと、見落としやすいポイント

**ツール**: Write

#### 9. /kaizen 実行

スキルやワークフローの改善点を振り返り、フィードバックする。

**ツール**: Skill (kaizen)

## 自己レビュー観点

### プラン（調査フェーズ）
- ナレッジベースの網羅性（関連する既存機能を見落としていないか）
- ヒアリングすべき不明点を見逃していないか

### requirements.md
- What/Why が明確か
- 要件に曖昧さが残っていないか
- 関連コードの調査は十分か
- 技術方針の候補は妥当か
- ドキュメントガイドライン（`.claude/rules/documentation.instructions.md`）の簡潔さの原則に従っているか

観点は内容に応じて調整。不明確な場合はユーザーにヒアリング。
