# OKR 定量化ガイド

Key Result を定量化する際のテクニック、測定基盤の分類、根拠の残し方。

## 定量化テクニック

定性的な目標を定量化する際、以下から適切なものを提案し、合意の上で採用する。

| テクニック | 適用場面 | 例 |
|-----------|---------|-----|
| 直接指標 | 測定可能な対象がある | 売上、ユーザー数、レスポンスタイム |
| プロキシ指標 | 直接測定が難しい | 「職場環境改善」→ eNPS スコア |
| マイルストーン達成率 | プロジェクト完了系 | 「新機能リリース」→ 4 機能中 X 機能完了 |
| 頻度・回数 | 行動・プロセス改善 | 「技術共有活性化」→ 勉強会を月 X 回開催 |
| 時間短縮 | 効率化系 | 「デプロイ高速化」→ リードタイム X 日 → Y 日 |
| 割合・比率 | 品質改善系 | 「バグ削減」→ 不具合率 X% 以下 |
| 満足度スコア | 主観的品質 | 「顧客体験向上」→ NPS X → Y |
| 二値判定 | 達成 / 未達が明確 | 「認証基盤刷新」→ 新基盤で全サービス稼働 |

### 原則

- 全 KR に現在値（ベースライン）と目標値を明記する
- 測定方法・データソースを併記する
- 測定基盤がない場合は A/B/C 分類に従って対処する（後述）
- 開発系の指標選定に困った場合は [SPACE フレームワーク](space-framework.md) を参照
- アウトプットではなくアウトカムで書く（[outcome-vs-output.md](outcome-vs-output.md)）

## 測定基盤 A/B/C 分類

KR 候補が出た時点で、測定基盤の状況を分類する。

| ランク | 状態 | 対応 |
|-------|------|------|
| 🟢 **A** | データソースが既にあり、今すぐ測定可能 | そのまま KR に採用 |
| 🟡 **B** | データソースがない、または計測体制が未整備 | 「Q1 中にベースライン計測を完了」を前提条件として記載。現在値は `[Q1 計測予定]` と記載し、目標値は暫定値を設定 |
| 🔴 **C** | 計測が現実的でない | 別のプロキシ指標に置き換えるか、KR 候補から除外 |

B ランクが 3 個以上ある場合、測定基盤の構築自体を 1 つの KR にすることを提案する。

## 定量化の根拠

定性的な目標を定量化した場合、その変換根拠を OKR ドキュメントに記載する。

```markdown
## 定量化の根拠

定性的な目標を定量化した場合の変換根拠:
- {KR}: {なぜこの指標・数値にしたか}
```

根拠がないと「なぜこの数値？」が追跡できず、中間見直しの判断ができない。
