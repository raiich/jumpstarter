# Fix Well

修正系の指示を受けたとき、曖昧なら解釈を 1 つに確定し、影響範囲・同類問題・整合性を先に洗い出して 1 回ですべて修正するスキル。指摘箇所のみ直すと「他にも同じパターンが」「呼び出し元が壊れた」「ドキュメントと矛盾」などの往復が発生する。

動作環境別の挙動は [environment.md](environment.md) を参照（実行前に一度読み込む）。

## フロー

1. **意図を解釈 → 対象を確定** — 指摘・フィードバックは一次表現（特定の語・1 箇所）のまま直さず、背後の原則・適用範囲をまず 1 文に言語化する。その上で何を / なぜ直すか（バグ / 命名整理 / 仕様変更 / 規約違反）を整理。スコープ・方針・成功基準のいずれかが一意に定まらない、または一次表現と原則レベルの解釈が割れ得るなら、先に下記「解釈の確定」で解釈を 1 つに固める
2. **影響範囲の洗い出し** — 指摘箇所を直す前に下表で洗い出す。網羅困難なら探索エージェントを並列に
3. **修正計画を集約** — 一次修正 / 同類修正 / 波及修正 / 影響なし（確認済み）を区別
4. **一括適用** — 1 つずつ待たない。ファイル数 5 以上、または変更が複数の概念にまたがる場合は「同じ意図の変更」単位（ファイル単位ではない）でグループ化
5. **完了報告** — ファイル/箇所ごとの変更内容と件数、確認したが該当しなかった範囲を1回で報告。「他に漏れは?」と聞き返さない

| 修正の種類 | 洗い出す対象 |
|---|---|
| 識別子変更（関数名・型名・設定キー） | 定義 + 全参照 + ドキュメント・コメント |
| パターンバグ修正 | 同じ書き方の他出現箇所すべて |
| API シグネチャ変更 | 呼び出し元 + 型定義 + ドキュメント例 |
| 仕様・規約変更 | 旧仕様前提のコード・テスト・ドキュメント |
| ドキュメント修正 | 同事実を述べた別ファイル |
| スタイル・記法統一（強調レベル・フォーマット・表記揺れ） | 変更箇所周辺の同種記述・類似ドキュメント |

## 解釈の確定（曖昧な場合）

スコープ・方針・成功基準が一意に定まらない依頼で、着手前に解釈を 1 つに確定する。「リファクタして」「整理して」「使いやすくして」など、対象が明示されていてもこれらが抜けやすい。

| 分類 | 何が不明か | 例 |
|---|---|---|
| 対象 | 何に対してか | 「これ直して」「整理して」 |
| スコープ | どこまで・どの粒度か | 「設定まわりを直して」 |
| 方針 | どう実現するか（可読性 / 性能 / 拡張性 等） | 「リファクタして」 |
| 制約 | 守るべき制約（互換性、ダウンタイム 等） | 「移行して」 |
| 成功基準 | 何が満たされれば完了か | 「使いやすくして」 |

1. **曖昧さの特定** — 解釈が分かれる箇所を 1〜3 個、上記分類で特定
2. **コードベース調査** — 既知パス/シンボルはファイル読み取り・コード検索、広範な探索は探索エージェントに範囲を指定して並列に。「コードを読めば分かること」をヒアリングで聞かない（[references/ask-the-user.md](references/ask-the-user.md) 第一原則）
3. **ヒアリング** — 調査で残った不足情報のみ構造化ヒアリング（1ラウンド最大 4 問）。[references/ask-the-user.md](references/ask-the-user.md) に従う。「自律的に進めて」と明示があっても複数候補があれば 1 ラウンドは許容する（承認を取るためではなく方針確定のため）。合理的解釈が成立する場合はスキップし、本作業の前提として明示する
4. **解釈の提示** — 下記形式で短く提示してから修正へ。ここで再度承認は取らない

```markdown
## 着手前の理解

- **やること**: [1〜2 文]
- **スコープ**: 含む / 含まない
- **方針**: アプローチと選択理由
- **前提**: ✅ 調査で確認した既存制約
- **推測**: ⚠️ 確認済み情報からの妥当な推測（あれば）
- **仮定**: ❓ ヒアリング不足で仮定した事項（あれば）
```

## 実践例

修正タスク: 「`err` 変数の未使用を修正」

**❌ 悪いアプローチ**:

1. 指摘された 1 箇所を修正
2. ユーザーに報告
3. ユーザー: 「漏れがあります」
4. 再検索して追加修正...

**✅ 良いアプローチ**:

1. コード検索で `err` の未使用パターンを全検索
2. 見つかった全ての箇所を一度に修正
3. ユーザーに報告: 「5 箇所の未使用 `err` 変数を修正しました」

## 注意

- 「影響なし」を明示するのは「他は調べていない」と「調べたが該当しなかった」を区別するため
- 「同じ意図の変更」と「ついでにきれいにしたい別の変更」は分ける。後者は別タスクとしてユーザー判断に委ねる（過剰修正で別問題を作らない）
- リネーム等の機械的置換は、置換後にビルド/型チェック/テストで参照漏れを検出するのが堅い
- 「他に修正が必要そうですが、修正しますか?」と聞き返さない（本スキル違反）
