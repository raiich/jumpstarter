# Fix

コードとドキュメントの修正に共通する原則とプロセス。

## 基本原則

- **修正方針を決めてから修正する** - 場当たり的に修正しない
- **影響範囲を確認してから修正する** - 関連箇所を全て把握する
- **同様の問題を一度に修正する** - 段階的な修正は避ける

## 修正フロー

### 1. 修正内容の理解

- ユーザーの修正指示を理解
- 修正対象の箇所を特定
- 修正による影響範囲を推測

### 2. 包括的検索の実施

**重要**: 同じ問題が他にもないか必ず確認

**手順:**

1. **パターン検索**: Grepで同じ問題が他にもないか検索
   - 例: エラーハンドリングの修正 → 同じパターンを全て検索
   - 例: 変数名の修正 → 同じ名前の使用箇所を全て検索

2. **関連ファイル確認**: 影響範囲の特定
   - 例: 関数名変更 → 全ての呼び出し箇所を確認
   - 例: 型定義変更 → 使用している全てのファイルを確認

### 3. 一括修正

見つかった全ての箇所を一度に修正
- 段階的な修正は避ける
- 同じ問題は1回の修正で全て解決

### 4. セルフレビュー

修正後、以下を確認：

- [ ] 全ての該当箇所を修正したか
- [ ] 修正内容は正しいか
- [ ] スタイルガイドラインに従っているか
- [ ] 関連するドキュメント・コードも更新が必要か

### 5. 報告

修正内容を簡潔に報告：
- 修正したファイルと箇所数
- 修正内容の概要
- 確認結果

## 実践例

```
修正タスク: "err変数の未使用を修正"

❌ 悪いアプローチ:
1. 指摘された1箇所を修正
2. ユーザーに報告
3. ユーザー: 「他にもないですか？」
4. 再検索して追加修正...

✅ 良いアプローチ:
1. Grep で "err" の未使用パターンを全検索
2. 見つかった全ての箇所を一度に修正
3. ユーザーに報告: 「5箇所の未使用err変数を修正しました」
```

## 目標

- 「他に漏れはないですか？」の質問をゼロにする
- 段階的な修正を避け、一度で完結した修正を実施
- ユーザーの時間を節約
