# Kaizen

セッション中の摩擦（ユーザー指摘・遠回り・ムダ・ストレス）を観察し、なぜなぜ分析で根本原因を「仕組みの不足」として言語化、エージェント設定に変更を適用して再発確率を下げる。

動作環境別の挙動は [environment.md](environment.md) を参照（実行前に一度読み込む）。

## 起動

- `/kaizen` — セッション全体（圧縮で失われた部分は対象外）
- `/kaizen <焦点>` — 焦点を1つ渡し、Phase 2 の絞り込み確認を省く

## 原則

- **現地現物**: 推測ではなく実際の会話・ツール呼び出しから事実を取る
- **なぜなぜ**: 5回目安・最低3回。「気をつけます」で止めない
- **仕組み化前の事実確認**: 「機能の存在」だけで案を出さない。**起動主体・代替手段・併用条件**を埋めてから設計に入る
- **仕組み化**: 個別修正で終わらせずエージェント設定に残す
- **小さく継続**: 1セッション最大3件

## フロー

### 1. 観察

セッションから事実を時系列で列挙し、推測と分離する。各事象に観察軸記号を付ける。

| 軸 | 観察対象 | シグナル例 |
|---|---|---|
| A | ユーザー指摘・修正依頼 | 「違う」「そうじゃない」 |
| B | Claude の遠回り・誤解・手戻り | 誤った仮説、無駄な探索 |
| C | ムダ・ムリ・ムラ | 重複読み込み、permission prompt 連発 |
| D | ストレスシグナル | 短い否定、苛立ち、トーン変化 |

### 2. 問題の選定

影響度 × 頻度で優先順位付けし最大3件に。候補2件以上なら構造化ヒアリングで対象を確認、1件なら確認なしで進む。

### 3. なぜなぜ分析と仕組み化案

各問題に対し、原因と対策を対で出す。

1. 事象を1文で記述
2. 「なぜ?」を最低3回、表層で止めず「設定、コンテキスト、進め方・プロセスの不備」に着地
3. 案を出す前に **目的と事実** を確定:
    - (a) 変更対象の目的（達成しないと成立しない要件）
    - (b) 動作環境の機能仕様（environment.md の「最新仕様の把握」）
4. environment.md の置き場所候補から最低2案を比較提示

下記を活用して対策案を出してもよい:

- [検証方法](../sketch-feature/references/verification-strategy.md)

### 4. 適用と引き継ぎ

選ばれた案を該当ファイルに適用し、変更サマリ（ファイル名 + 要点）を1回出力。
ユーザーが次セッションで観察すべきポイントを 1〜2 行で出力（ファイルには残さない）。

出力例: [output-example.md](output-example.md)

## 注意

- 表層の謝罪・推測での断定・なぜなぜ1〜2回止め・適用に至らない分析は本スキル違反
- 既存設定の前提を崩さない、組み込みスキル/コマンドと重複する提案をしない
- 摩擦の対策に承認ゲート・確認ステップを増やすのは慎重に。
  ゲート1回のコストは出力後の修正指示1回とほぼ同等で、見合うのは修正指示が多発する・不可逆な場合のみ。それ以外はゲートを設けず出力するのをデフォルトにする
- 常時コンテキストに乗る手段（常時ロードされる指示書など）を増やすのは慎重に。
  コンテキスト圧迫の代償があるので、必要時のみ参照される形（スキル化・フック化など）で代替できないか先に検討
