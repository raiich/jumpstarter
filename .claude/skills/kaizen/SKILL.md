# Claude Code 改善コマンド

conversation.log からフィードバックを抽出し、learnings.md に反映した上で、コードベースのギャップ分析も含めて Claude Code 設定を改善します。

## 実行手順

### フェーズ1: フィードバック抽出

1. **ログ読み込みと判定**
   - `.local/claude/conversation.log` を読み込む
   - ログが存在しない、または空の場合はこのフェーズをスキップし、フェーズ2へ進む

2. **フィードバック抽出**
   - ユーザーからの指摘、修正依頼、方針指示、好み・判断基準を抽出
   - 単なる作業指示（「○○して」）は除外し、再利用可能な知見のみ抽出
   - 既に `.local/claude/learnings.md` に記載済みの内容は除外

3. **learnings.md への追記**
   - 抽出結果が0件の場合は learnings.md を更新せず、フェーズ2へ進む（learnings.md に既存の知見があればそれを基に改善可能なため）
   - ログも存在しないか空で、かつ learnings.md も空の場合は処理を終了する
   - `.local/claude/learnings.md` の該当カテゴリに追記:
     - **方針**: 開発方針、意思決定の背景
     - **考慮すべき観点**: 重視する観点、判断基準
     - **進め方のコツ**: 効果的なやり方、段取り
     - **注意点**: 避けるべきこと、見落としやすいポイント
   - 各項目は1行で簡潔に記述
   - 追記内容をユーザーに提示し、承認後に更新

   **例: フィードバック抽出の判断**

   ログ: 「Design Doc にシーケンス図を入れてほしかった」
   → ✅ 抽出する（再利用可能な知見: Design Doc に処理フローがある場合はシーケンス図を含める）

   ログ: 「auth.go の L42 を修正して」
   → ❌ 抽出しない（一回限りの作業指示）

4. **ログクリア**
   - 承認・更新完了後、`.local/claude/conversation.log` をクリア（空ファイルにする）

### フェーズ2: Claude Code 設定の改善

1. **フィードバック分析**
   - `.local/claude/learnings.md` を読み込む
   - 記録されたフィードバックから改善可能なパターンを特定

2. **コードベースのギャップ分析**
   - 以下を簡易的に把握する（深掘りしすぎない）:
     - 言語・フレームワーク・ディレクトリ構造
     - テスト構成、CI/CD、主要な外部依存
     - ドキュメント構成、Makefile / タスクランナーの有無
   - 現在の `.claude/` 設定と突き合わせ、カバーされていない領域を特定:
     - **ルールの不足**: コードベースにパターンがあるのにルール化されていない
     - **スキル・エージェントの不足**: 繰り返し作業や専門的視点が欠けている
     - **手順・自動化の不足**: セットアップ、ビルド、テスト、デプロイ手順の欠如
     - **設定の不整合**: 既存設定同士の矛盾やコードベース実態との乖離

3. **Claude Code 機能の確認**
   - 以下のドキュメントを確認（キャッシュあり）:
     - `https://code.claude.com/docs/en/claude_code_docs_map.md`
     - `https://code.claude.com/docs/en/changelog.md`
   - **キャッシュ手順**:
     1. `.local/claude/cache/` のキャッシュファイルを Read で読み込む
        - `claude_code_docs_map.md`、`changelog.md`
     2. ファイル先頭のフロントマター `fetched_at: YYYY-MM-DD` と現在日付を比較し、**7日以内** ならそのまま使用
     3. ファイルが無い or 7日超過の場合のみ curl で取得し、以下の形式で Write:
        ```
        ---
        fetched_at: YYYY-MM-DD
        ---
        （取得内容）
        ```
     4. **重要**: 取得内容は要約・リネームせず、そのまま保存する
   - フィードバックパターンを解決できるClaude Code機能を特定
   - 既存の設定で活用できていない機能がないかチェック

4. **改善案の作成**
   - フィードバック・ギャップ分析・Claude Code機能を組み合わせた改善案を作成
   - 以下の設定ファイルを対象：
     - `.claude/settings.local.json`（個人設定。テンプレートリポジトリのため優先）
     - `.claude/settings.json`（テンプレート共有設定）
     - `.claude/agents/`
     - `.claude/skills/`
     - `.claude/commands/`
     - `.claude/rules/`（実体は `.github/instructions/`。編集は実体側で行う）

5. **改善の実施**
   - 改善案をユーザーに提示
   - 承認後、設定ファイルを更新

## 出力フォーマット

```
## フィードバック分析結果

### 主要なフィードバック
- [フィードバック1: 頻度X回]
- [フィードバック2: 頻度Y回]

### コードベースのギャップ
- [ギャップ1: 種別 - 根拠]
- [ギャップ2: 種別 - 根拠]

### 改善提案

#### 提案1: [タイトル]
- **対象**: [ファイル名]
- **内容**: [変更内容]
- **効果**: [期待される改善]

#### 提案2: [タイトル]
...

### Claude Code 最新機能の活用
- [機能名]: [活用方法]
```

## 注意事項

- 既存設定を破壊しないよう注意
- 変更前にバックアップを推奨
- ユーザー承認後に変更を実施
- 既存の組み込みスキル・コマンド（`/simplify` 等）と重複する提案をしない
- **learnings.md は kaizen 専用の中間データ**。通常セッションへのコンテキスト注入（SessionStart フック等）は提案しない。改善は `.claude/` 設定ファイルに反映する
- **learnings.md とメモリは独立**。フィードバック抽出時、メモリに保存済みかどうかは気にせず learnings.md に記録する。両者は目的が異なる（learnings.md は kaizen の改善入力、メモリは通常セッションの行動指針）
