# Sketch Feature

## 1. 概要

機能の追加・改修を、ビルドが通る実コードのスタブで起こす。「なぜ」は薄いドキュメントに、「何を・どう作るか」はスタブで表現し、インターフェース整合を型検査に委ねる。未実装は `panic("implement me (generated by sketch-feature skill)")` / `t.Fatal` で残し、実装が進めば消える（別の状態管理は持たない）。

成果物は次の 2 フェーズ。

1. 設計スケッチ → `design.md`（薄いドキュメント、検証方法を含む）＋ 設計スタブ（コード）
2. テストスケッチ → テストスタブ（コード）

動作環境別の挙動とテンプレートは [environment.md](environment.md) / [sketch-templates.md](sketch-templates.md) を参照。参照する観点は §5。

## 2. 原則

- **ヒアリング**: 設計スケッチで 1 回限り。`design.md` の不足情報を一括解消する。前提が崩れたときのみ設計スケッチに戻る
- **ビルド維持**: シグネチャを起こした時点でビルド・型検査が通ること。型・インターフェース・シグネチャは実型で書き、本体は未実装を示す形（言語慣用の中断: 例外・パニック等）にする
- **テストは失敗させる**: テストスタブは各ケースで明示的に失敗させ未実装を可視化する。コミットすると CI も失敗する前提で扱う
- **コメント方針**: public な型・フィールド・メソッド・インターフェースには doc コメント（契約・設計意図）を書く。private・実装内部はコメント最小（コードだけで意図が分かるなら書かない）。専用マーカーは使わない
- **承認ゲート**: `design.md` の ❓ 仮定項目の検証のみユーザー確認する
- **提示で終了**: テストスケッチ完了時点でスタブを提示して終了する。未実装の追跡は `panic` / `t.Fatal` と落ちるテストに任せ、別管理ドキュメントは作らない

## 3. 起動と再開

### 3.1 起動条件

スコープで判別する（複数コンポーネント・複数ファイルにまたがる、または設計判断が必要な変更）。設計・テストを実コードのスタブで表現したい場合に選ぶ。ドキュメント主体で設計・合意したい場合、差分を 1 文で説明できる個別修正の場合は対象外。

### 3.2 再開

原則として最初から通す。再開する場合は、どこから再開するか・既存成果物を活かすか作り直すかをユーザーにヒアリングしてから進める。

## 4. フェーズ

各フェーズは「インプット → アウトプット → 確認」で進める。

### 4.1 設計スケッチ（薄い design.md ＋ 設計スタブ）

- **インプット**: 既存コード・ドキュメントの把握（テスト基盤・既存パターン・依存関係）と、`design.md` の不足情報を一括解消する構造化ヒアリング（1 回限り）。不足分だけ追加調査する
- **アウトプット**（テンプレは [sketch-templates.md](sketch-templates.md)）
    - 薄い `design.md`: 背景・目的・技術的アプローチ（選択理由・代替案）・検証方法（`verification-strategy.md` 観点で選定）・考慮事項のみ。インターフェース詳細は書かない
    - 設計スタブ: public な型・フィールド・メソッド・インターフェースのシグネチャを実型で書き、それぞれに doc コメントで契約・設計意図を書く。本体は未実装、private な内部実装は実装時に追加する
- **確認**: ビルド・型検査が通ること。`design.md` 観点でセルフレビュー（§5）

### 4.2 テストスケッチ

- **インプット**: 既存テスト基盤の調査（[testing.md](references/testing.md) の「既存テスト基盤の流用」）
- **アウトプット**: テストスタブ。ケースごとに独立したブロックを列挙し（テーブルにまとめない）、各ケースで明示的に失敗させる。確かめる内容を Given/When/Then のコメントで示す
- **確認**: ビルドが通ること。`testing.md` 観点でセルフレビュー（§5）

ここで終了し、スタブを提示する。

## 5. 参照する観点

セルフレビュー時の観点は固定リストを読み上げず、[derive-perspectives.md](references/derive-perspectives.md) の手順でその場で導出し、[missable-checklist.md](references/missable-checklist.md)（自力で出にくい取りこぼし項目）と突き合わせる。インターフェース仕様は薄い `design.md` ではなく設計スタブ側で評価する。

導出を補助する手順ファイル:

- 検証手段の選定 → [verification-strategy.md](references/verification-strategy.md)
- テストスケッチ・テスト品質 → [testing.md](references/testing.md)
- エージェント特有の盲点 → [agent-pitfalls.md](references/agent-pitfalls.md)
