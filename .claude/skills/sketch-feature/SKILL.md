# Sketch Feature

## 1. 概要

機能の追加・改修を、ビルドが通る実コードのスタブで起こす。「なぜ」は薄いドキュメントに、「何を・どう作るか」はスタブで表現し、インターフェース整合を型検査に委ねる。未実装は言語慣用の中断（`panic` / `t.Fatal` 等）で残し、実装が進めば消える（別の状態管理は持たない）。

スケッチの対象は公開境界（外部から参照されるシグネチャ）に絞り、内部実装や不確実な要素はコード化しない。レビューを要点に集中させ、余計なリファクタと不確実な実装を避けるため。

成果物は、薄い `design.md`（なぜ・検証方法）と、設計スタブ・テストスタブ（コード）。手順は §4。

動作環境別の挙動とテンプレートは [environment.md](environment.md) / [sketch-templates.md](sketch-templates.md) を参照。参照する観点は §5。

## 2. 原則

- **ヒアリング**: 設計スケッチで 1 回限り。不足情報を一括解消する。前提が崩れたときのみ設計スケッチに戻る
- **ビルド維持**: シグネチャを起こした時点でビルド・型検査が通ること。型・インターフェース・シグネチャは実型で書き、本体は未実装を示す形（言語慣用の中断: 例外・パニック等）にする
- **テストは失敗させる**: テストスタブは各ケースを対象言語の慣用（例外送出・fail 呼び出し等）で明示的に失敗させ未実装を可視化する。空のテスト・常に通る本体にしない。コミットすると CI も失敗する前提で扱う
- **コメント方針**: 公開するシグネチャには doc コメント（契約・設計意図）を書く。インラインコメントとして擬似コードやTODOコメントを書く
- **承認ゲート**: `design.md` の ❓ 仮定項目の検証のみユーザー確認する
- **提示で終了**: テストスケッチ完了時点でスタブを提示して終了する。未実装の追跡は `panic` / `t.Fatal` と落ちるテストに任せ、別管理ドキュメントは作らない。残した ❓ は理由と判断材料を `design.md` に明記し、後続の実装が別セッションでも辿れるようにする

## 3. 起動と再開

### 3.1 起動条件

スコープで判別する（複数コンポーネント・複数ファイルにまたがる、または設計判断が必要な変更）。設計・テストを実コードのスタブで表現したい場合に選ぶ。ドキュメント主体で設計・合意したい場合、差分を 1 文で説明できる個別修正の場合は対象外。

### 3.2 再開

原則として最初から通す。再開する場合は、どこから再開するか・既存成果物を活かすか作り直すかをユーザーにヒアリングしてから進める。

## 4. フェーズ

各フェーズは「インプット → アウトプット → 確認」で進める。

### 4.1 設計スケッチ（薄い design.md ＋ 設計スタブ）

- **インプット**: 既存コード・ドキュメントの把握（テスト基盤・既存パターン・依存関係）と、不足情報を一括解消する構造化ヒアリング（1 回限り）。不足分だけ追加調査する
- **アウトプット**（テンプレは [sketch-templates.md](sketch-templates.md)）
    - 薄い `design.md`: why（背景・目的・選択理由・代替案）・スコープ・検証方法（`verification-strategy.md` 観点で選定）・横断的考慮事項のみ（仕様・手順・受け入れ条件はコード側）。各記述の信頼度は [confidence-marks.md](references/confidence-marks.md) の ✅ / ⚠️ / ❓ で示す
    - 設計スタブ: 公開境界（外部から参照される型・構造体・フィールド・メソッド・関数・インターフェース・変数）のシグネチャだけを実型で書き、doc コメントで契約・設計意図を書く。公開境界かは「別パッケージから参照・言及されているか」で判定する。内部的にのみ参照される型は、そのパッケージに特徴的・中核実装であっても徹底して省略する（private 構造体やその上のメソッド・private 関数を sketch で起こさない。「中核だから」「将来再利用しそう」は公開理由にならない）。内部実装や、公開境界か確信が持てないものはシグネチャに起こさず、doc コメントと本体内の TODO・インラインコメントで表す。一度起こしたシグネチャを外す（削除・降格）のは呼び出し側を壊す手戻りだが、TODO に留めたものを実装時に起こすのは非破壊なので、不確実な sketch では確信できる公開境界だけを狭く起こす
- **確認**: ビルド・型検査が通ること。セルフレビュー（§5）

### 4.2 テストスケッチ

- **インプット**: 既存テスト基盤の調査（[testing.md](references/testing.md) の「既存テスト基盤の流用」）
- **アウトプット**: テストスタブ（テンプレ・例は [sketch-templates.md](sketch-templates.md)）。ケースごとに独立ブロックで列挙する（テーブルにまとめない: ケースごとに Given/When/Then と失敗を独立可視化するため）
- **確認**: ビルドが通ること。セルフレビュー（§5）

ここで終了し、スタブを提示する。

## 5. 参照する観点

セルフレビュー時の観点は固定リストを読み上げず、[derive-perspectives.md](references/derive-perspectives.md) の手順でその場で導出し、[missable-checklist.md](references/missable-checklist.md)（自力で出にくい取りこぼし項目）と突き合わせる。インターフェース仕様は薄い `design.md` ではなく設計スタブ側で評価する。

導出を補助する手順ファイル:

- 検証手段の選定 → [verification-strategy.md](references/verification-strategy.md)
- テストスケッチ・テスト品質 → [testing.md](references/testing.md)
- エージェント特有の盲点 → [agent-pitfalls.md](references/agent-pitfalls.md)

## 6. コメントの寿命

- **公開境界の doc コメント**: 実装後も残す（API ドキュメント）
- **未実装マーカー** (`panic("implement me (generated by sketch-feature skill)")`・テストの `t.Fatal("not implemented (generated by sketch-feature skill)")`): 残っている限り未実装。実装時に必ず消える（未実装の判定はこれが根拠）
- **`// TODO`**: ただのコメント。スタブ時の実装ヒント用に書いたものは実装時に消すが、将来やりたい改善メモとして書いたものは残してよい（両者の書式は区別しない。判別できないものは残す）
- **実装詳細のコメント**: 「コードだけ読んで意図が分かるなら削除、分からない意図（所有権・前提・非自明なトレードオフ）だけ残す」で判断する
