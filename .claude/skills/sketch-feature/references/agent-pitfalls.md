# コーディングエージェント特有のメタ観点

学習データの偏りや暗黙の慣習に引きずられた挙動を抑えるための観点。あらゆるタスクに横断適用する。

## 不確実性の表明

「分からない」「未確認」「仮定」と「確認済み」を区別して表明する。確認可能なことは確認し、未確認のまま進む場合は仮定であると明示する。
信頼度マークの記法は [sketch-templates.md](../sketch-templates.md) を参照する。

## 既存コードとの整合性確認

命名規則・ディレクトリ構成・テストの書き方・エラー処理パターンは、まず既存コードを参照する。学習データ由来の「一般的に良いとされる書き方」を無意識に持ち込まないかを点検する。
ただし、既存パターンにセキュリティ・正確性・廃止予定 API の問題がある場合は流儀を踏襲せず、理由を明示して別の判断を取る。整合性は手段であり、コード品質を下げてまで守る目的ではない。

## 変更前の現状把握

変更を加える前に、対象コードの現在の挙動を理解する。失敗ケース・期待される入出力・既存テストの有無・隣接コードからの依存を確認してから手を入れる。現状を再現する手順か、現状を表明するテストが手元にあるかを点検する。「とりあえず書き換える」は、後で挙動の差分が説明できない原因になる。

## 変更の影響範囲確認

シンボル名・ファイルパス・設定キー・公開 API を変更する際は、参照箇所を全件確認する。1 ファイルの変更が他箇所の前提を崩していないか、コード検索で実証する。「たぶん他にはない」は根拠にならない。

## スコープの逸脱抑制と変更粒度

依頼された範囲を超えた変更を加えない。1 つの変更単位（コミット・PR）は 1 つの目的に絞る。バグ修正のついでのリファクタ、書き換えに紛れた挙動変更、機会を捉えた抽象化は、依頼外作業として分離する。大規模変更はレビュー可能な粒度に段階分割する。

## テスト実行と結果の検証

テストを書いた・通したと報告する前に、実際にテストランナーを実行し出力を確認する。コードの差分から「通るはず」と推論せず、出力を読む。期待値を実装側に合わせて書き換えていないか、アサーションが評価される経路を通っているか、テストがスキップ扱いになっていないかを点検する。

## テスト・型・静的解析の過信回避

型・テスト・リンタが通っても「仕様が満たされている」とは限らない。これらは「壊れていない」ことの一部しか保証しない。仕様レベルの確認（ユーザーストーリー・受け入れ条件・手動検証）を別途行う。逆に、警告を抑止して通すことを目的化しないようにする。

## 完了報告のバイアス回避

- 「動きました」「pass しました」で失敗・部分実装・既知の不具合を覆い隠さない
- 完了報告には未対応項目・残課題・前提崩れの可能性を率直に列挙する
