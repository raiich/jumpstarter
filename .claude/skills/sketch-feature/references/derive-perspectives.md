# 観点の導出手順

レビュー・修正タスクで「何を確認すべきか」を、固定リストの読み上げではなく対象コードからその場で導出する手順。
導出後に [missable-checklist.md](missable-checklist.md) と突き合わせ、自力では出にくい取りこぼしを防ぐ。

## 1. 対象を分類する

変更対象が該当するレンズを選ぶ（複数可）。各レンズが見るべき観点を方向づける。

| レンズ | 該当条件 |
|---|---|
| ネットワーク API | HTTP/REST・GraphQL・gRPC・WebSocket を他者に提供 |
| ライブラリ・基盤 | 利用者がコードに組み込む。公開境界・互換性責任を負う |
| サービス・運用 | 常駐プロセス・デプロイ・監視対象 |
| データストア | RDBMS / NoSQL / KVS / ファイル永続化 |
| UI / フロントエンド | ユーザーが直接操作する画面 |

## 2. データフローを引く

- 外部入力 (source) → 内部処理 → 出力・特権操作・永続化 (sink) を列挙し、信頼境界線を引く
- 境界を越える値ごとに「検証・正規化・出力エンコードはどこか」を問う
- 公開 API 表面（外から呼べる関数・型・エンドポイント）を境界として明示する

## 3. リソースとライフサイクルを追う

- 確保するもの（FD・接続・ロック・メモリ・task）ごとに、全経路（正常・異常・キャンセル）で解放されるか
- 所有権（誰が確保し誰が解放するか）が API シグネチャから読めるか

設計レビューでは、凝集度・結合度・依存方向・公開境界を同じ source/sink・所有権の枠組みで点検する。

## 4. 自己反証で間引く

- 各観点に「最小再現入力」と「観測可能な指標（ログ・メトリクス・スタック）」を 1 行で書けるものだけ採用する
- 書けない思いつき・一般論は落とす

## 5. 取りこぼしを突き合わせる

- 導出した観点を [missable-checklist.md](missable-checklist.md) の該当レンズと照合し、自分が出さなかった取りこぼし項目を拾う
- 各レンズで「該当なし」も明示する（沈黙と見落としを区別する）
