# 失敗時の根本原因分析

テスト失敗・バグ・予期せぬ挙動に対して、場当たり的な修正を避けるためのプロセス。

## 1. 直接原因の特定

エラーメッセージ、スタックトレース、失敗した assertion から直接の原因を特定する。

## 2. 上流の原因を探る

- 直接原因のさらに上流を探る（なぜそうなったか？）
- 症状の出た場所が本当の原因か確認する
- 同じ問題が他の場所でも起きていないか、コード検索でパターンを横展開で確認する

## 3. 修正候補を複数検討

- この修正は場当たり的でないか？根本原因に対処しているか？
- 他のテストや機能に影響しないか？
- 設計レベルの見直しが必要ではないか？

## 4. 原因の分類と対応

| 分類 | 対応 |
|---|---|
| 実装のバグ | 原因箇所（症状の箇所とは限らない）を修正し再テスト |
| テスト設計のミス | テストケース設計を見直し、テストスタブを更新 |
| 設計の問題 | `design.md` を更新して設計スケッチに戻る（→ SKILL.md §4.1） |
| 判断不可 | ユーザーにヒアリング |

## エスカレーション

- **修正が 2 回失敗したら**: 同じアプローチを繰り返さず、別の原因仮説やアプローチを検討する
- **3 回失敗したら**: ユーザーに相談する
