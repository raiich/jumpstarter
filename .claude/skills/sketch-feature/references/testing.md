# テストの観点

テストケース設計・テストコード・テストレビューに共通する具体的観点。

観点の導出は [derive-perspectives.md](derive-perspectives.md)、取りこぼしの安全網は [missable-checklist.md](missable-checklist.md)、エージェント特有の盲点は [agent-pitfalls.md](agent-pitfalls.md) を併用する。本ファイルはテスト設計・実装・レビュー時の具体的観点に絞る。

## 最小セット

対象タスクに関わらず、まず以下を確認する。網羅は非機能要件がある場合や、最小セットでリスクが残ると判断した場合に行う。

- **振る舞いベース**: 実装の写しではなく「ユーザーの操作 → 結果」を検証しているか（→ 「テスト品質」節）
- **要件からの導出**: 機能要件 1 つに対しテストケースが対応しているか、漏れがないか（→ 「網羅性」節）
- **False positive 防止**: 通過するように見えるが評価されていない assertion がないか（→ 「False positive 防止」節）
- **Assertion の妥当性**: 期待値が独立した根拠から導出されているか（→ 「Assertion の妥当性」節）

「非機能テスト」「既存テスト基盤の流用」は該当する条件が成立したときのみ参照する。

## 網羅性

- 機能要件からの導出漏れがないか（要件 → テストケースの対応）
- 正常系・異常系・エッジケースのバランス
- 重要なユーザーシナリオがカバーされているか
- エクスポートされた関数・メソッドにテストが存在するか

## テスト品質

- **振る舞いをテストしているか**: 「ユーザーの操作 → 結果」を検証している。定数値・配列サイズ・enum メンバ数・プロパティ値のアサーションはテストではない
- **バグ検知器か、変更検知器か**: プロダクションコードの正しいリファクタリングでテストが壊れる、または値の変更にテストが機械的に追従するだけなら、そのテストは実装に結合しており価値が低い
- **テストの起点は機能要件か**: 機能要件（「〜したら〜になる」の受け入れ条件）から導出している。実装詳細（型定義、モジュール構造）から導出していない
- **public API を通してテストしているか**: 内部関数・内部状態を直接テストしない。モジュールの利用者視点で書く
- **テストが使用例のドキュメントになっているか**: テストを読めばそのモジュールの使い方がわかる
- **モックが複雑になっていないか**: モックが膨らむのはテスト対象の責務が大きすぎるサイン。テスト設計かプロダクション設計を見直す

## False positive 防止

- ループ内 assertion の対象が空でないことが保証されているか
- 条件分岐で assertion がスキップされるパターンがないか
- 出力から期待値を導出する循環ロジックがないか

## Assertion の妥当性

- 期待値が独立した根拠から導出されているか
- 許容幅（tolerance）が適切か（広すぎてテストが意味をなしていないか）
- 上限・下限の片方しかチェックしていないケースがないか

## 引数の一致

- 関数呼び出しが全必須引数を渡しているか
- 型チェックが通っても、ランタイムで undefined になる引数がないか

## テスト構造

- 各テストが独立しているか（共有状態、実行順序への依存がないか）
- Arrange-Act-Assert が明確に分離しているか
- モックのセットアップ／クリーンアップが適切か
- テスト名が振る舞いを表しているか
- **DAMP > DRY**: ヘルパーで隠しすぎず、各テスト内で何をテストしているかが読み取れることを優先

## 非機能テスト

機能テストでは検出できない品質を、専用の手法で確認する。各サブ手法は「適用トリガー」に該当しなければスキップする。

### 並行性テスト

- 適用トリガー: 共有状態がある / 複数スレッド・プロセス・タスクが同時に動く / 順序保証や排他制御が要件 / データ競合の可能性がある
- 競合・デッドロック・メモリ可視性違反を検出する負荷テスト・並行テスト
- スケジューラ介入（goroutine yield、`thread.sleep` 注入、決定的スケジューラ）
- 順序保証の検証（イベントログを取って後から確認）

### 性能テスト・ベンチマーク

- 適用トリガー: レイテンシ・スループット・メモリ使用量の数値目標がある / 回帰検出が必要 / 公開ライブラリで利用者が性能特性を必要とする
- レイテンシ・スループット・メモリ使用量の数値目標に対する継続的計測
- ベンチマークの落とし穴（最適化による消去・ウォームアップ不足・GC 揺らぎ）の回避
- ベースライン比較（リグレッション検出）

### Property-based testing

- 適用トリガー: 入出力の不変条件が形式化できる / 入力空間が広く例示テストだけではカバー困難 / 対象がデータ変換・パーサ・アルゴリズム
- 入力をランダム生成して不変条件を検証
- 失敗ケースの最小化（shrinking）に対応したフレームワーク（QuickCheck / Hypothesis / fast-check 等）

### Fuzz testing

- 適用トリガー: 信頼境界の入力を扱う / パーサ・デコーダ・プロトコル実装 / セキュリティ要件がある
- パーサ・デコーダ・外部入力境界で実施
- カバレッジガイド型 fuzzer（libFuzzer / AFL++ / Go の `go test -fuzz`）
- クラッシュ最小化と再現テストの自動追加

### Mutation testing

- 適用トリガー: テストスイートの実効性を定量評価したい / カバレッジ高でもバグが流出する経緯がある / ライブラリ等で品質指標が必要
- テストスイートが本当にバグを検出できるかを「ミューテーション（プロダクションコードの変異）」で検証
- mutation score を品質指標として参照（PIT / Stryker / mutmut 等）

## 既存テスト基盤の流用

テストケース設計・実装前に、既存のテスト用インフラを調査して再利用する。

### 調査対象

- テスト対象に関連するモック・スタブ・テストヘルパー（`test/`, `mock/` 配下など）
- setUp / tearDown ユーティリティ
- fixtures やテストデータ

### 原則

- 再利用可能なものがあれば優先して活用する
- プロダクションコードのファクトリ・ヘルパーを再利用する（test-helpers に再実装しない）
- テスト固有コード（時間制御（fake clock 等）・Mock）のみを新規ヘルパーに置く
- 新たに必要な場合（例: 遅延制御や実行順序の観測）は理由を明示する

### 判定方法

テストファイルの import を確認する。プロダクションコードと同じ処理が test-helpers に書かれていないかをコード検索で確認する。
