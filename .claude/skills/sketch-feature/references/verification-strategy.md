# 検証方法の観点

タスクの「正しさをどう確かめるか」を決める際の観点集。仕様設計で `design.md` の「検証方法」を埋めるときに参照する。
検証手段が決まれば実装方針は逆算できる、効果の大きい設計判断。

## 完了の定義

- 何が成立すれば「正しい」と言えるかを 1〜3 文で言語化する
- 抽象表現（「使いやすく」「綺麗に」「速く」）は不可。観測可能な命題に置き換える
  - 「使いやすく」→「N 手順で目的達成できる」
  - 「速く」→「P50 < 200ms, P99 < 1s」
  - 「綺麗に」→「lint pass / 関数長 N 行以下 / 循環的複雑度 M 以下」
- 「含まないこと」も明示する（スコープ外の振る舞いに対する誤検出を防ぐ）

## 検証手段の存在

| 手段 | 適する対象 | コスト |
|---|---|---|
| 自動テスト | ロジック・API・データ変換 | 低（一度書けば高速反復） |
| Lint / 型検査 | 表層の規約・型整合 | 低 |
| コマンド出力検証 | スクリプト・CLI ツール | 低〜中 |
| ビジュアル比較 | UI・レイアウト・配色 | 中（スクリーンショット差分） |
| 計測値 | 性能・リソース消費 | 中〜高（負荷環境が必要） |
| 手動確認 | 上記で代替できない体感頼みの確認 | 高（毎回ユーザー時間を要する） |

- 既存の手段で済むなら新規作成しない（既存テスト基盤の流用観点は [testing.md](testing.md)）
- 既存に手段がなければ「検証を作る」を最初のタスクに含める

## 対象種別ごとの典型手段

- **ライブラリ・基盤ソフト**: コードレビュー＋テストケースのレビュー（網羅性・境界・異常系）＋自動テスト実行
- **画面描画を伴うアプリ（Web GUI・ゲーム等）**: 自動操作（Web GUI は Playwright 等、ゲームは自動プレイ）＋スクリーンショット＋画像差分の検証

対象別の取りこぼしやすい観点は [missable-checklist.md](missable-checklist.md) を併用する。

## 検証コスト

- 1 サイクル（変更 → 検証結果取得）が秒〜分単位で回るか確認する
- 手動確認しかない場合、開発中の試行錯誤コストが大きく膨らむ。環境固有の非対話実行（→ [environment.md](../environment.md)）での部分自動化や、最小再現スクリプトでの代替を検討する
- E2E や統合テストしかない領域では、対象付近にユニット相当の検証点を追加する

## 検証の独立性

- テストが実装の写しになっていないか（観点は [testing.md](testing.md) の「テスト品質」「Assertion の妥当性」）
- 自分が書いたコードのセルフレビューは盲点が残る。独立視点（別エージェント / 別セッション / レビュー専用エージェント）を併用する

## 根本原因か症状か

- エラーメッセージの抑制・try-catch で握りつぶす・テストの skip で済ませていないか
- 「ビルドが通る = 正しい」と混同していないか（型が通ってもランタイムで落ちる引数不一致は典型）
- 失敗時はまず再現テストを書き、それが失敗することを確認してから修正する（分析手順は [root-cause-analysis.md](root-cause-analysis.md)）

## 検証不能なケース

- 検証手段がどうしても存在しない場合は「検証不能」を明示する
- そのとき選ぶ代替:
  - スコープを縮め、検証可能な部分だけ先に出す
  - レビュー段階を厚くする（独立視点でのコードレビュー / ペアレビュー）
  - 段階的ロールアウト（カナリア・フィーチャーフラグ）で本番観測を検証に代える
- 「検証不能」と言わずに進めない

## ビジュアル変更の扱い

- 「綺麗にして」「いい感じに」は検証不能になりがち。参照画像・現状スクリーンショット・期待差分の言語化を入力に含める
- 変更前後のスクリーンショットを並べる差分検証を組み込む（環境固有の手段がある場合は [environment.md](../environment.md) を参照、なければ手動のスクリーンショット比較で代替）
