# テンプレート（薄い design.md / 設計スタブ / テストスタブ）

`sketch-feature` の各成果物のテンプレートと記法。コード例は Go だが、方針は言語非依存（ビルド・型検査が通る／テストが明示的に失敗する、を満たせばよい）。

## 薄い design.md

「なぜ」と検証方法を残す。インターフェース・データ構造・処理フローの詳細は設計スタブ（コード）に置く。

```markdown
# Design Doc: [名前]

## 背景・目的

## 技術的アプローチ（選択理由、代替案）

## スコープ
- 含むもの / 含まないもの

## 検証方法
- 自動テスト / Lint・型 / コマンド出力 / 計測値 / 手動 / 検証不能（+ 理由）

## 考慮事項
- セキュリティ・性能・互換性など、コードのコメントに収まらない横断的判断

## ファイル
- 主要ソース・テストのパス（コードへの誘導が他で十分なら省略可）
```

書かないもの: インターフェース仕様・シグネチャ・データ構造・処理手順（→ 設計スタブ）。

## 設計スタブ（コード）

public な型・フィールド・メソッド・インターフェースのシグネチャを実型で書き、ビルド・型検査を通す。private な内部実装は実装時に追加する。

- public な型・フィールド・メソッド・インターフェースには doc コメントを書く。契約・設計意図（前提・所有権・エラー方針など）を載せ、名前の言い換えで終わらせない
- private・実装内部のコメントは最小（コードだけで意図が分かるなら書かない）
- 本体は言語慣用の中断（Go なら `panic("implement me (generated by sketch-feature skill)")`）で未実装を示す。これが残っている限り未実装で、実装時に消える。`// TODO: 何をするか` は実装ヒントとして併記してよい（将来の作業メモ用 TODO と書式上の区別はしない）

Go の例:

```go
// Resource は Registry が保持する解放可能なリソースです。
type Resource interface {
	io.Closer
	// Name は Registry 内でリソースを一意に識別する名前を返します。
	Name() string
}

// Registry は確保済み Resource を名前で保持します。
// 解放責務は Registry にあり、Close で全 Resource を解放します。
type Registry struct{}

// Add は res を登録します。同一 name の既存 Resource は呼び出し側が解放済みである前提です。
func (r *Registry) Add(res Resource) error {
	// TODO: 重複 name の検出と登録
	panic("implement me (generated by sketch-feature skill)")
}

// Close は登録済みの全 Resource を解放し、最初に発生したエラーを返します。
func (r *Registry) Close() error {
	// TODO: 全 Resource を解放し、最初のエラーを返す
	panic("implement me (generated by sketch-feature skill)")
}
```

doc コメントは契約・意図を載せる（`Add` の「呼び出し側が解放済みである前提」など、シグネチャから読めない情報）。`// TODO` は doc コメントとは別に置く。未実装ヒントとして書いたものは実装時に消す。

## テストスタブ（コード）

`func Test〜` で対象（関数・メソッド）ごとにまとめ、ケースを `t.Run` で列挙する。各ケースで明示的に失敗させ、確かめる内容を Given/When/Then のコメントで示す。

Go の例:

```go
func TestRegistry_Add(t *testing.T) {
	t.Run("新規 name を登録できる", func(t *testing.T) {
		// Given: ...
		// When:  ...
		// Then:  ...
		t.Fatal("not implemented (generated by sketch-feature skill)")
	})

	t.Run("重複 name はエラーを返す", func(t *testing.T) {
		t.Fatal("not implemented (generated by sketch-feature skill)")
	})

	t.Run("Close 後の Add はエラーを返す", func(t *testing.T) {
		t.Fatal("not implemented (generated by sketch-feature skill)")
	})
}
```

ケースは `design.md` の要件・設計スタブの責務から導く。実装時に `t.Fatal` を外し、Given/When/Then を実コードに置き換える。

## 信頼度マーク（✅ / ⚠️ / ❓）

薄い `design.md` の要件・設計項目に情報源の確かさを示す（コードのコメントには付けない）。

| マーク | 意味 | 条件 |
|---|---|---|
| ✅ | 確認済み | ユーザー発言・コード・ドキュメントから直接確認した事実 |
| ⚠️ | 推測 | 確認済み情報からの妥当な推測 |
| ❓ | 仮定 | 情報源なし／実装前にユーザー確認が必要 |

❓ が多いときはヒアリングで解消できないか検討する。

## コメントの寿命

- **public 面の doc コメント**: 実装後も残す（API ドキュメント）
- **未実装マーカー** (`panic("implement me (generated by sketch-feature skill)")`・テストの `t.Fatal("not implemented (generated by sketch-feature skill)")`): 残っている限り未実装。実装時に必ず消える（未実装の判定はこれが根拠）
- **`// TODO`**: ただのコメント。スタブ時の実装ヒント用に書いたものは実装時に消すが、将来やりたい改善メモとして書いたものは残してよい（両者の書式は区別しない）
- **private・実装内部のコメント**: 「コードだけ読んで意図が分かるなら削除、分からない意図（所有権・前提・非自明なトレードオフ）だけ残す」で判断する
