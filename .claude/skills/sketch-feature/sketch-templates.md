# テンプレート（薄い design.md / 設計スタブ / テストスタブ）

`sketch-feature` の各成果物のコピー元テンプレートと記入の最小注記。コード例は Go だが、方針は言語非依存（ビルド・型検査が通る／テストが言語慣用の手段で明示的に失敗する、を満たせばよい）。ルール・規約（公開境界の判定・コメント方針・コメントの寿命・信頼度マークなど）は SKILL.md と references/ が正本。

## 薄い design.md

コードとコメントで表現できるものは書かない（仕様・手順・受け入れ条件はコード側）。残すのは why（背景・目的・選択理由・代替案）・スコープ・検証方法・横断的考慮事項のみ。

```markdown
# Design Doc: [名前]

## 背景・目的

## 技術的アプローチ（選択理由、代替案）

## スコープ
- 含むもの / 含まないもの

## 検証方法
- 自動テスト / Lint・型 / コマンド出力 / 計測値 / 手動 / 検証不能（+ 理由）

## 考慮事項
- セキュリティ・性能・互換性など、コードのコメントに収まらない横断的判断

## ファイル
- 主要ソース・テストのパス（コードへの誘導が他で十分なら省略可）
```

## 設計スタブ（コード）

公開境界のシグネチャだけを実型で書き、ビルド・型検査を通す。公開境界の判定と内部実装の扱いは SKILL.md §4.1 に従う（内部実装の実体は実装時に private として追加）。
