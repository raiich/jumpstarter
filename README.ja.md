# Jumpstarter - Claude Code の設定テンプレート

[ [English](README.md) | **日本語** ]

## 概要

Claude Code から期待する出力を得るには、適切なプロンプトを与える必要があります。
また、Rules / Skills / Hooks といった設定の整備にもコストがかかります。

Jumpstarter は、clone して使い始めるだけでこれらの課題を解消し、開発そのものに集中できるようにします。
Claude Code の利用で**ユーザーが頑張らない**ことを目指します:

- **長いプロンプト不要** — コードベース調査・要件のヒアリング・自己レビューを適切に実行する仕組みを用意しています。
- **凝った仕組みは後回し** — MCP やマルチエージェントは後回し。Rules / Skills をメインに使います。
- **継続的改善も手軽** — 改善点やベストプラクティスにもとづき、設定を継続的に改善していけます。

さらに、エージェントを安全に自律実行するための **dev container** も同梱しています。外部通信を許可した宛先だけに制限し、Node / Python / Go / .NET などの言語ランタイムも揃えているため、隔離された環境ですぐに開発が始められます。

## 使い方

メインの使い方は、自分のリポジトリで `jumpstart` を実行し、同梱の dev container を再利用することです。jumpstarter 自体を開いて使ったり、`.claude/` だけをコピーすることもできます。

### dev container で使う

jumpstarter を一度 clone し、対象リポジトリで `jumpstart` を実行して dev container を再利用します。`jumpstart` は jumpstarter の `.devcontainer` を対象リポジトリへシンボリックリンクし（コピーではなくリンクなので、テンプレート側の更新が自動で反映されます）、bind mount で jumpstarter の `.claude/` をそのリポジトリ直下に見せます。

```bash
git clone https://github.com/raiich/jumpstarter.git

# 対象リポジトリで実行（jumpstarter/bin に PATH を通すか、フルパスで呼ぶ）
cd /path/to/other-repo
jumpstart
```

実行後、対象リポジトリを dev container で開き直してください（VS Code: **Reopen in Container**。Docker と Dev Containers 拡張が必要）。リンクはマシン依存のパスを指すので commit しないでください。許可ドメインは `.devcontainer/gateway/allowed-domains.acl`、転送ポート（既定で開発サーバー用の 5173）は `devcontainer.json` で調整できます。

> **補足.** Claude の認証・設定は共有ボリューム `claude-code-config` に保存されるため、ログイン状態や設定変更（`/kaizen` 等）はこのコンテナを使う全プロジェクトに反映されます。

#### 兄弟ディレクトリも使える

コンテナは対象リポジトリの*親*ディレクトリをマウントするため、ワークスペースに加えて同階層の兄弟ディレクトリ（自作ライブラリなど）もコンテナ内から参照・編集できます。ローカルのライブラリをそのまま使えて便利です。ただし読み書き可能なため、secrets を含むリポジトリは同じ親に置かないでください。

#### jumpstarter 自体を開いて使う

jumpstarter を clone してそのまま dev container で開けば、`jumpstart` なしでも利用できます。テンプレート自体の試用・開発に向きます。

### .claude/ だけコピーして使う

dev container を使わず、既存プロジェクトに `.claude/` をコピーするだけでも使えます。手軽ですが、ネットワーク隔離やランタイムは付きません。

```bash
git clone https://github.com/raiich/jumpstarter.git

# 既存プロジェクトに .claude/ をコピー（日本語版。英語版は整備中）
cp -R jumpstarter/.claude "${YOUR_PROJECT}/"

claude
```

### Claude Code で開発する

セットアップできたら、`claude` を起動して普段どおり開発するだけです。同梱の Rules / Skills が自動で働くため、特別なプロンプトは要りません。特定の場面では、以下のワークフローも呼び出せます。

### 新機能をスケッチする

機能追加に完璧なプロンプトは要りません。最小限の指示から、独立した設計書ではなく**ビルドが通るスケッチ**を起こします。「なぜ」は薄いドキュメントに、「何を・どう作るか」はスタブに表現し、インターフェース整合は型検査に委ねます。

`/sketch-feature` スキルは、コードベースを調査しつつ要件をヒアリングし、次を生成します:

- public な型・シグネチャ（doc コメント付き、ビルドは通る）
- 未実装のうちは明示的に失敗するテストケース

```
/sketch-feature CLIにverboseオプションを追加したい
```

薄い `design.md` は `.local/docs/features/[名前]/` に、スタブは実ソースツリーに保存されます。あとはスタブを埋めて実装します。

### 設定を継続的に改善する

開発体験が良くならないのは、日々のフィードバックが設定に反映されないことが多いためです。以下の仕組みで、設定の改善も頑張らずに回せます。

`/kaizen` は、セッションで起きた問題を観測し、なぜなぜ分析で根本原因を特定して、設定の改善で再発を防ぎます:

```
/kaizen                       セッション全体を観測対象に
/kaizen <問題の記述>           特定の問題に焦点を絞る
```

「なぜこうなった?」と尋ねた時にも発動し、表面的な謝罪ではなく根本原因と再発防止策で応えます。

`/import-best-practices` は、ネット上のベストプラクティスを取り込みます:

```
/import-best-practices   https://... の記事のベストプラクティスを取り込みたい
```

これらにより、ルール・スキル・フックがプロジェクトに合わせて洗練されていきます。

## その他の設定

`.claude/` ディレクトリの各要素が連携して「頑張らない」を実現します。

### 頑張らないための Rules

毎回指示しなくても、一定の品質基準で動作するためのルール群です (`.claude/rules/`)。

- 冗長な出力を頑張って読み解く必要はありません → **writing-style**
- 迎合・未検証の主張を頑張って訂正する必要はありません → **no-sycophancy**

### 頑張らないための Skills

明示的に呼び出すワークフローに加え、日々の摩擦を減らすスキル群です (`.claude/skills/`)。

- 中途半端な修正や曖昧な変更依頼で頑張る必要はありません → **fix-well**（曖昧なら着手前に解釈を 1 つに確定し、関連箇所も整合性を保って修正）
- 1 つの視点で頑張って考え抜く必要はありません → **hyper-think**（並列マルチエージェントでレビュー / 発想）

## カスタマイズ

テンプレートをコピーした後、プロジェクトに合わせて調整できます。

- **スキル・ルールの追加**: `.claude/skills/` や `.claude/rules/` にファイルを追加
- **自動改善**: `/kaizen` でセッションの問題から根本原因を分析し再発防止策を設定に反映、`/import-best-practices` で外部記事のベストプラクティスを取り込み
